# Spec 6.8 — Cambios programados del partido

> Subfase del Plan Maestro: **6.8 — Cambios programados: minuto + jugador out + jugador in + razón**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-29 (bosquejo)

---

## 1. Contexto

El staff prepara, antes del partido, un plan de cambios: minuto N, sale X, entra Y, motivo (rotación, físico, táctico, gestión de tarjeta, etc.). Esa lista es un input fundamental tanto para reaccionar bajo presión en F7 como para revisar a posteriori si el plan se cumplió. Sin esta pieza, F7 obliga al coach a improvisar mientras opera el cronómetro.

## 2. Objetivos

- Crear, editar y borrar entradas `(minuto, player_out_id, player_in_id, reason)` para una alineación.
- Lista ordenada por minuto, visible en una sección lateral u overlay del editor de F6.
- Validar coherencia mínima: `player_out_id` debe estar en el campo en ese momento (titular o entrado por cambio anterior); `player_in_id` debe estar en banquillo.
- El minuto puede ser 0 (cambio pre-pitido inicial — útil cuando un titular se lesiona en calentamiento).

## 3. Fuera de alcance

- **NO** se ejecutan los cambios automáticamente durante el partido. La lista es un **plan**, no una orden. F7 importará la lista al iniciar partido y le irá dando paso al coach con confirmación manual ("aplicar este cambio ahora").
- **NO** se sugieren cambios automáticamente (basados en stats, minutos jugados, tarjetas). Esa capa de recomendación entra en una Ola posterior (probable F13 o más allá).
- **NO** se calcula impacto táctico ("este cambio rompe la formación X"). Validación dura ya prevista en F6.7 (jugador en una sola zona); la coherencia táctica es decisión del coach.

## 4. Modelo de datos afectado

```sql
create table public.planned_substitutions (
  id            uuid primary key default gen_random_uuid(),
  lineup_id     uuid not null references public.lineups(id) on delete cascade,
  minute        smallint not null check (minute >= 0 and minute <= 120),
  player_out_id uuid not null references public.players(id),
  player_in_id  uuid not null references public.players(id),
  reason        text not null check (reason in (
    'rotacion',
    'fisico',
    'tactico',
    'gestion_tarjeta',
    'planificado',
    'otro'
  )),
  notes         text check (notes is null or char_length(notes) <= 300),
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),

  constraint planned_subs_distinct_players
    check (player_out_id <> player_in_id)
);

create index planned_subs_lineup_minute_idx
  on public.planned_substitutions (lineup_id, minute);
```

RLS: misma policy que `lineup_positions` (cualquier miembro del club lee, el manager del lineup escribe).

## 5. UX

- Sección lateral del editor de F6: "Cambios programados (3)".
- Botón "+ añadir cambio": dropdown jugador campo → dropdown jugador banquillo → input minuto → select reason → textarea notes opcional.
- Lista ordenada por minuto ascendente. Cada fila editable inline.
- Si un cambio referencia un jugador que ya no está en campo/banquillo (ej. el coach lo movió a "fuera"), la fila se marca en rojo con icono de warning. No se borra automáticamente — el coach decide.

## 6. Validaciones server-side

- Trigger BEFORE INSERT / UPDATE verifica:
  - `player_out_id` aparece en `lineup_positions` con `location IN ('field', 'bench')` del mismo `lineup_id`.
  - `player_in_id` aparece en `lineup_positions` con `location='bench'`.
  - Excepción: una vez aplicado en F7, el chequeo se desactiva (necesita columna `applied_at` o flag, decisión en implementación).

## 7. Tests

- Vitest puro: ordenación + dedupe + flag de warning si el cambio referencia un jugador fuera de campo/banquillo.
- pgTAP: trigger rechaza `player_out_id` que no está en el `lineup_positions` del mismo lineup; rechaza `player_out_id = player_in_id`.
- Smoke: planificar 3 cambios (min 60, 70, 80). Editar el segundo a min 65. Lista se reordena. Borrar el tercero. Lista queda con 2.
