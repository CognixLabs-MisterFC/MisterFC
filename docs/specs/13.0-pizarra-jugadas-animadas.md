# Spec 13.0 — Pizarra de jugadas animadas (playbook + modo presentación)

> Fase del Plan Maestro: **F13 — Pizarra táctica 2D con animación (modo iPad)**.
> Estado: ☑ cerrada (2026-06-27). El modelo de jugadas evolucionó después de esta spec: la **serie JR** (ADR-0019, #229–#232) lo reorganizó de team-scoped a **banco del club + ciclo de aprobación + `team_plays`** (playbook por equipo + compartir con familia). El §5 (modelo de datos) de esta spec describe el diseño *original* y queda **superado por ADR-0019**; el contrato/editor/animación/fullscreen siguen vigentes. **13.8 (export) descartado y eliminado del roadmap.**
> Autor: Iker Milla
> Fecha de creación: 2026-06-20

## 1. Contexto

F11 dejó el **contrato del diagrama** (`@misterfc/core`, Zod puro, **frame-extensible**), el editor `<PitchBoard>` (reducer puro `pitchEditorReducer`) y el renderer read-only `<DiagramView>`. F11B montó una **pizarra táctica EFÍMERA** (sin persistencia) sobre tres fondos (en blanco / ejercicio / once real). Ambas fases dejaron el terreno preparado para F13 **a propósito**:

- El contrato exporta **solo la escena estática** `Diagram = { version, field, elements[] }` y **NO comprometió** el tipo de los frames — "esa forma la decide F13" (comentario en `packages/core/src/diagram/diagram.ts`).
- Cada elemento tiene un **`id` estable** (unicidad garantizada por `superRefine`) y la posición es **separable**: `elementAnchors(el): DiagramPoint[]` expone los puntos-ancla geométricos (flecha→`[from,to]`, línea→`points`, zona/punto→`[pos]`). **El `Diagram` estático de F11 = un frame.**

F13 construye **encima**: envuelve la escena en **frames** para **animar** una jugada, la **persiste** en una biblioteca del equipo (**playbook**), permite **compartirla** con jugadores/familias y añade un **modo presentación a pantalla completa** (vestuario / iPad). F13 **NO rediseña** el contrato del diagrama, el editor ni el renderer: los **reutiliza**.

**Frontera con F11B (explícita)**: F13 es el **autor de jugadas** — diseño persistente, animado por frames, biblioteca del equipo, presentación en vestuario. La **presentación EN VIVO sobre la alineación real** durante el partido/entreno ya es **F11B** (efímera, sin guardar). F13 es **diseño, no directo**.

**Dependencias previas cubiertas**: F11 cerrada (contrato + `<PitchBoard>`/`<DiagramView>` + reducer), F11B cerrada (pizarra + `renderField`), F12 (patrón `visibility staff/team` + vista read-only de jugador/familia), F2.10 (patrón listado), F1.7 (helpers RLS), F1.4 (capabilities por dominio).

## 2. Objetivos

- El entrenador **diseña una jugada animada**: una secuencia de **frames** (cada frame = una escena del diagrama) con la **timeline** para añadir/duplicar/reordenar/borrar frames.
- **Reproducción** de la jugada **interpolando posiciones** entre frames (play/pause/scrub/loop/velocidad).
- **Playbook del equipo**: biblioteca persistente de jugadas (`team_id`), listado con filtros + CRUD.
- **Compartir con el equipo**: jugadores y familias ven la jugada read-only (memorizarla).
- **Modo presentación** a pantalla completa para el vestuario (iPad/tablet), **solo lectura**, con fallback que funcione en iOS Safari.

## 3. Decisiones cerradas (D1–D8)

| # | Decisión |
|---|---|
| **D1** | **Scope TEAM**. Una jugada pertenece a un equipo: `team_id NOT NULL` (FK `teams`). Es el **playbook del equipo** (a diferencia de `exercises`, que es club-wide). `club_id` se denormaliza para la RLS. |
| **D2** | **Creación DIRECTA** + **`visibility 'staff'\|'team'`** (default `staff`, patrón sesiones F12 / `lineups`). **Sin** ciclo `draft→proposed→published` ni aprobación del admin. Trigger simple (owner forzado a `auth.uid()`, owner/club inmutables, `updated_at`, forma del jsonb). En `'team'` → jugadores **y** familias del `team_id` la ven **read-only**. |
| **D3** | **Frame = escena COMPLETA**. La jugada es `{ version, field, frames: DiagramScene[] }`: cada frame lleva su `elements[]` íntegro (reutiliza el validador de elemento de F11). La animación casa elementos **por `id`** entre frames consecutivos. Añadir/borrar elementos por frame es trivial (están o no en el array). (Alternativa de deltas/keyframes descartada por menor reutilización.) |
| **D4** | **Tempo por frame**: cada frame tiene `duration_ms` opcional; si falta, se aplica un **default global** (`DEFAULT_FRAME_MS`). La duración es el tiempo de **transición hacia** el siguiente frame. |
| **D5** | **Interpolación = solo POSICIONES**. Se interpolan los `elementAnchors` por `id` con **lerp lineal**. Propiedades discretas (rol, color, label, estilo de flecha/línea, tamaño) **NO se interpolan**: saltan en el límite de frame. Elemento presente en un frame y no en el adyacente → **aparece/desaparece (fade)**. Easing/curvas/rotación interpolada = **backlog**. |
| **D6** | **Export (13.8) DESCARTADO** (2026-06-27). F13 = **13.0–13.7**. Se elimina del roadmap (no diferido): el caso de uso no aporta. |
| **D7** | **Navegación**: el **playbook** cuelga del **hub Entrenamientos** (junto a Ejercicios / Sesiones / Pizarra) + **acceso desde la ficha del equipo**. |
| **D8** | **Modo presentación = SOLO LECTURA** en v1 (reproduce/presenta; **no** edita a pantalla completa — editar = el editor normal). |

> **Versionado**: `PLAY_VERSION = 1`, **independiente** de `DIAGRAM_VERSION`. El contrato de jugada puede evolucionar (p.ej. easing) sin tocar el del diagrama, y viceversa.

## 4. Contrato de la jugada animada (core — 13.1a)

Vive en `@misterfc/core` (`src/diagram/play.ts` o `src/plays/`), **encima** del contrato del diagrama, sin modificarlo. Zod puro, sin DOM/BD/React. Reutiliza `diagramElementSchema` / `fieldSchema` / `elementAnchors`.

```ts
export const PLAY_VERSION = 1 as const;
export const MAX_FRAMES = 60;             // tope de seguridad por jugada
export const DEFAULT_FRAME_MS = 1000;     // D4: tempo por defecto entre frames
export const MIN_FRAME_MS = 100;
export const MAX_FRAME_MS = 20000;

// Un frame = una escena del diagrama (D3) + su duración de transición (D4).
// `elements[]` reutiliza el validador de F11; `field` es de la jugada (abajo),
// no del frame (el campo no cambia entre frames).
const frameSchema = z.object({
  elements: z.array(diagramElementSchema).max(MAX_DIAGRAM_ELEMENTS),
  duration_ms: z.number().int().min(MIN_FRAME_MS).max(MAX_FRAME_MS).optional(),
});
export type PlayFrame = z.infer<typeof frameSchema>;

export const playSchema = z.object({
  version: z.literal(PLAY_VERSION),
  field: fieldSchema,                      // campo común a todos los frames
  frames: z.array(frameSchema).min(1).max(MAX_FRAMES),
});
export type Play = z.infer<typeof playSchema>;   // el jsonb `play` de la BD

export function parsePlay(input: unknown): SafeParseReturnType<unknown, Play>;
export function isPlay(input: unknown): input is Play;
export function emptyPlay(field?: Partial<DiagramField>): Play; // 1 frame vacío
```

> Cada frame valida ids únicos **dentro del frame** (mismo `superRefine` que `diagramSchema`). Un mismo `id` **debe** repetirse entre frames para que el elemento sea "el mismo" y se interpole; un `id` nuevo = elemento que aparece.

### 4.1 Interpolación (core — base de 13.3)

Helper **puro y testeable** que, dado el array de frames y un tiempo de reproducción, devuelve una **escena estática** `Diagram` lista para pintar con `<DiagramView>`:

```ts
// t en milisegundos desde el inicio de la jugada. Devuelve la escena en ese
// instante: posiciones interpoladas (lerp de elementAnchors por id) entre el
// frame actual y el siguiente; props discretas tomadas del frame de origen;
// elementos que solo existen en uno de los dos lados → opacidad 0→1 (fade).
export function sceneAtTime(play: Play, t_ms: number): Diagram;

// Duración total = suma de duration_ms (o DEFAULT_FRAME_MS) de los frames 1..n-1.
export function playDurationMs(play: Play): number;
```

- **Casado por `id`**: para cada elemento con `id` presente en el frame de origen y en el destino, se interpolan sus `elementAnchors` (lerp lineal componente a componente); el resto de campos se copian del **origen** (D5).
- **Aparición/desaparición**: un `id` solo en origen → fade-out durante la transición; solo en destino → fade-in. (La opacidad la consume `<DiagramView>` vía un campo de presentación, no se persiste.)
- **Último frame**: a `t ≥ playDurationMs` se muestra el último frame estático.
- Sin estado, sin `Date.now()`: el componente de reproducción (13.3/13.4) lleva el reloj (rAF) y llama a `sceneAtTime`.

## 5. Modelo de datos (`plays` — 13.1b)

Calca el patrón de `exercises`/`sessions`: `owner_profile_id` forzado por trigger, `club_id` denormalizado, inmutabilidad de owner/club, índices por `(club_id, …)`. **Migración nueva**; no toca tablas existentes.

```sql
create table public.plays (
  id                uuid primary key default gen_random_uuid(),
  owner_profile_id  uuid not null references public.profiles(id) on delete cascade,
  club_id           uuid not null references public.clubs(id)    on delete cascade,
  team_id           uuid not null references public.teams(id)    on delete cascade,  -- D1
  name              text check (name is null or char_length(name) between 1 and 120),
  description       text,
  play              jsonb not null,                 -- Play (D3); validación autoritativa = parsePlay en app
  visibility        text not null default 'staff'   -- D2
                      check (visibility in ('staff', 'team')),
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now()
);

create index plays_team_idx  on public.plays (team_id, updated_at desc);
create index plays_club_idx  on public.plays (club_id);
create index plays_owner_idx on public.plays (owner_profile_id);
```

### 5.1 Trigger (simple — D2)
- `INSERT`: `owner_profile_id := auth.uid()`; `club_id` coherente con `team_id` (derivado del equipo).
- `UPDATE`: `owner_profile_id` / `club_id` / `team_id` **inmutables**; `updated_at := now()`.
- **Forma ligera del jsonb** (la autoritativa es `parsePlay` en la app): `play` es objeto, `play->'frames'` es array no vacío y `<= MAX_FRAMES`. (Espeja la validación ligera del diagram en `exercises`.)

### 5.2 RLS (scope TEAM + visibility — D1/D2)
Helper de autoría reutilizable (¿quién crea/edita jugadas de un equipo?): admin/coord del club, o **staff activo del `team_id`** (`team_staff`), o capability de entrenamientos. Espeja `user_can_create_sessions` / `user_can_create_exercises`.

- **SELECT**:
  - `visibility = 'staff'` → autor **o** staff del equipo **o** admin/coord del club.
  - `visibility = 'team'` → además, **jugadores y familias** del `team_id` (read-only). Reutiliza el patrón de visibilidad de `sessions` (F12.4).
- **INSERT**: `owner_profile_id = auth.uid()` **y** puede crear jugadas en ese `team_id` (helper).
- **UPDATE/DELETE**: autor **o** admin/coord (gate = RLS). Sin transiciones de estado (no hay ciclo, D2).

**pgTAP** de la RLS (scope team + visibility), verificado contra el remoto (F15.8: pgTAP fuera de CI).

## 7. Modo presentación / pantalla completa (13.0 — primer paso)

Shell de **pantalla completa** montable **sobre la pizarra y el diagrama existentes** (sin modelo, sin frames) → entrega valor ya y de-riska la UX iPad.

- **Fullscreen API** (`element.requestFullscreen`) **CON fallback a overlay CSS** (posición fija a pantalla, `100dvh`, z alto, oculta el chrome) para **iOS Safari**, que **no** soporta `requestFullscreen` en elementos sueltos → tiene que funcionar en móvil/tablet **sí o sí**.
- **La pizarra sigue EDITABLE en grande**; el **visor de diagramas** (ejercicio) entra en fullscreen **de visualización** (read-only).
- **Salir**: tecla **Esc** + **botón** de cerrar. **Gestos avanzados (swipe entre frames) NO en 13.0** — van con el playback/presentación (**13.7**).
