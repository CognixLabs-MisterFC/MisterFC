## 4. Modelo de datos

### 4.1. Decisión D1: tabla `events` única (no jerarquía de subtipos)

Justificación: las 5 tipologías (`training`, `match`, `tournament`, `friendly`, `other`) comparten el 80% de los campos y se diferencian por unos pocos opcionales (`opponent_name` en match/friendly). Hacer una tabla por tipo o un STI con tabla auxiliar es overengineering para Ola 1 y complica las queries de calendario que cruzan tipos. Decisión: tabla única + campo `type` validado por CHECK + campos opcionales documentados.

### 4.2. Decisión D1: `team_id` y `category_id` "at most one set", `club_id` denormalizado

> **Nota terminológica**: en una versión previa de esta spec llamé al constraint "XOR". Es incorrecto: XOR estricto excluiría el caso "ambos NULL", que es precisamente el evento a nivel club. El comportamiento real es **"at most one is set"**, que se traduce a:
> ```sql
> check (team_id is null or category_id is null)
> ```
> Permite las 3 combinaciones válidas: (team, NULL) → equipo · (NULL, category) → categoría · (NULL, NULL) → club. Excluye solo (team, category).


```
events.club_id      not null  ← multi-tenant, raíz
events.team_id      nullable  ← evento del equipo concreto
events.category_id  nullable  ← evento de la categoría (todos los equipos)
                              ← ambos NULL = evento del club (todos)
```

Por qué `club_id` aunque sea derivable de `team`/`category`:

- Sin `club_id` directo, los filtros multi-tenant requieren JOIN doble (`events → teams → categories` o `events → categories`) en RLS y en cada query. Postgres lo hace, pero el coste en lectura es real.
- Para eventos de club (sin equipo ni categoría) no hay JOIN posible: `club_id` es la única forma.
- Costes: una columna `uuid` + un trigger de coherencia. Trade-off claro.

Por qué `team_id` xor `category_id`:

- Modelo de producto: un evento es del **equipo X**, de **toda la categoría Y** (típico: torneo prebenjamín) o del **club entero** (gala anual). No tiene sentido que sea simultáneamente "del Benjamín A *y* de toda la categoría Prebenjamín".
- Si se necesitara en futuro publicar el mismo evento a varios equipos, el modelo se extiende con tabla puente; por ahora no se cubre.

### 4.3. Decisión D2: recurrencia Opción A (parent + children explícitos)

**Por qué A, no B (RRULE), no C (sin recurrencia)**:

- **C (sin recurrencia)** descartada porque los entrenamientos en fútbol base son periódicos casi por definición: martes + jueves 18:00–19:30 toda la temporada. Sin recurrencia, el entrenador crea ~80–120 entrenamientos a mano por equipo y temporada. Fricción enorme que mata la adopción.
- **B (RRULE estilo iCalendar)** descartada porque cubre casos que el dominio no tiene ("tercer martes excepto julio", "cada 2 semanas alternas en años bisiestos"). El coste de implementar RRULE correctamente (DST, exclusiones, edición selectiva) supera con creces el valor. Las librerías existentes (`rrule.js`) añaden ~60KB al bundle.
- **A** elegida: cubre el 95% del caso real (entreno semanal en días fijos) con un esquema trivial y un generador puro de ≤80 líneas.

**Forma de A**:

- `parent_event_id` nullable en la misma tabla `events`: el parent tiene `parent_event_id = NULL` y `recurrence_rule jsonb` rellena; los children tienen `parent_event_id = parent.id` y `recurrence_rule = NULL`. Por construcción, una serie es de profundidad 1.
- `recurrence_rule` jsonb con esquema rígido validado por Zod:

```ts
{
  freq: 'weekly',                  // único valor soportado en F3
  interval: 1,                     // 1 = todas las semanas. Permitido 1–4.
  by_weekday: number[],            // 0=lunes ... 6=domingo. Min 1, max 7. Ordenado.
  count?: number,                  // número de SEMANAS en la serie (no de hijos). count XOR until.
  until?: string,                  // ISO date 'YYYY-MM-DD'.
}
```

- **Límite duro: 52 SEMANAS de serie**, no 52 hijos. Un equipo entrena habitualmente 2–3 días por semana × 30–40 semanas/temporada = 60–120 ocurrencias por serie. Eso debe entrar sin problema.
  - Si `count` está presente: `count` representa el número de **semanas** generadas (no hijos). El generador produce `count × by_weekday.length` ocurrencias en total. Valido `1 ≤ count ≤ 52`.
  - Si `until` está presente: el generador itera hasta que la siguiente semana exceda la fecha. Valido `until ≤ first_event_date + 365 días`.
  - Para mensajería UI, el form muestra el cálculo en directo: "108 ocurrencias generadas (36 semanas × 3 días)" para que el usuario entienda qué va a crear antes de guardar.
- Generación al guardar: al crear un evento con `recurrence_rule`, una server action **transactional** inserta el parent y N children (todos con el `parent_event_id` apuntando al parent). Al editar el parent con "esta y futuras" se borran los children con `starts_at >= ahora` y se regeneran. "Solo esta" edita el child directamente.
- Borrado: "esta instancia" borra el child; "esta y futuras" borra children con `starts_at >= esta`; "toda la serie" borra parent (los children caen por `on delete cascade`).
- Limitación aceptada: si el parent se mueve a otra fecha, los children no se "arrastran" automáticamente — se asume que mover el parent es "editar esta y futuras" lo cual regenera.

### 4.4. Migración SQL (skeleton)

```sql
-- 20260530000000_events.sql (F3.1)

create table public.events (
  id                uuid primary key default gen_random_uuid(),

  -- aislamiento multi-tenant + targeting
  club_id           uuid not null references public.clubs(id)      on delete cascade,
  team_id           uuid     null references public.teams(id)      on delete cascade,
  category_id       uuid     null references public.categories(id) on delete cascade,

  -- clasificación
  type              text not null check (type in (
    'training', 'match', 'tournament', 'friendly', 'other'
  )),

  -- cabecera
  title             text not null check (char_length(title) between 1 and 200),
  notes             text,

  -- ventana temporal
  starts_at         timestamptz not null,
  ends_at           timestamptz,
  all_day           boolean not null default false,

  -- localización (opcional)
  location_name     text check (location_name is null or char_length(location_name) between 1 and 160),
  location_address  text check (location_address is null or char_length(location_address) between 1 and 240),

  -- match / friendly
  opponent_name     text check (opponent_name is null or char_length(opponent_name) between 1 and 120),

  -- recurrencia (Opción A)
  parent_event_id   uuid references public.events(id) on delete cascade,
  recurrence_rule   jsonb,

  -- audit
  created_by        uuid not null references public.profiles(id),
  created_at        timestamptz not null default now(),
  updated_at        timestamptz not null default now(),

  -- D1: at most one of (team_id, category_id) set. Permite (team, NULL), (NULL, category), (NULL, NULL).
  constraint events_target_at_most_one check (team_id is null or category_id is null),

  -- ends_at >= starts_at cuando esté presente
  constraint events_window_valid check (ends_at is null or ends_at >= starts_at),

  -- recurrence_rule solo en parents; children no la llevan
  constraint events_recurrence_only_in_parent check (
    parent_event_id is null or recurrence_rule is null
  )
);

create index events_club_starts_idx        on public.events (club_id, starts_at);
create index events_team_starts_idx        on public.events (team_id, starts_at) where team_id is not null;
create index events_category_starts_idx    on public.events (category_id, starts_at) where category_id is not null;
create index events_parent_idx             on public.events (parent_event_id) where parent_event_id is not null;

create trigger events_set_updated_at
  before update on public.events
  for each row execute function public.set_updated_at();

alter table public.events enable row level security;
```

**Trigger de coherencia `club_id ↔ team_id / category_id`** (mismo patrón que `invitations_assert_team_same_club`):

```sql
create or replace function public.events_assert_target_same_club()
returns trigger language plpgsql security definer set search_path = public as $$
declare
  derived_club uuid;
begin
  if new.team_id is not null then
    select c.club_id into derived_club
    from public.teams t join public.categories c on c.id = t.category_id
    where t.id = new.team_id;
    if derived_club is distinct from new.club_id then
      raise exception 'team belongs to a different club' using errcode = '23514';
    end if;
  end if;
  if new.category_id is not null then
    select c.club_id into derived_club from public.categories c where c.id = new.category_id;
    if derived_club is distinct from new.club_id then
      raise exception 'category belongs to a different club' using errcode = '23514';
    end if;
  end if;
  return new;
end$$;

create trigger events_target_same_club_check
  before insert or update of club_id, team_id, category_id on public.events
  for each row execute function public.events_assert_target_same_club();
```

### 4.6. RLS — Decisión D6

**Helpers existentes que reutilizamos**:

- `user_role_in_club(club_id) → text`
- `user_has_capability_in_club(club_id, capability) → bool`
- `user_is_staff_of_team(team_id) → bool` ← F2.6, pieza clave

**SELECT — abierto a cualquier miembro del club (decisión deliberada de Ola 1)**:

```sql
create policy events_select_member on public.events
  for select to authenticated
  using (public.user_role_in_club(club_id) is not null);
```

> **Esto es una simplificación deliberada de Ola 1.** Asumimos que los eventos son **semi-públicos dentro de un club** (título, fecha, lugar; sin datos sensibles). El filtro "jugador ve solo eventos de sus equipos y eventos de club" se aplica en la **query del Server Component**: es **UX, no seguridad**.
>
> Implicación operativa: un jugador autenticado puede consultar vía REST/RPC con cualquier `team_id` del club y recuperar eventos a los que no pertenece. **Esto es intencional en Ola 1** y se documenta como known-issue `F3-rls-events-visibilidad` para endurecimiento en **F14 (RGPD para menores)** mediante un helper `user_can_see_event(event_id) → bool`.
>
> Razones para no endurecer ahora:
> - Beta cerrada con un solo club piloto: superficie de "abuso" prácticamente nula.
> - No expone datos sensibles (medical/RGPD vive en `players.medical_notes` con su propia RLS estricta de F2.2).
> - Simplifica la matriz de policies (sin esta apertura habría 4–5 ramas SELECT distintas por rol).
> - Permite que admin/coord vean **todo el club** sin policies separadas.
> - Endurecimiento posterior es localizado: cambiar solo la policy SELECT + añadir helper. No requiere refactor del modelo.

Este comentario aparece **también** verbatim como bloque comentario al inicio del archivo SQL de la migración, para que cualquier futuro lector entienda el porqué sin tener que cruzar a `docs/`.

**INSERT / UPDATE / DELETE — por rol + capability**:

```sql
-- helper: ¿puede el user gestionar este evento?
-- Devuelve true si:
--   - admin_club o coordinador del club, o
--   - entrenador_principal del team_id (o de cualquier team si event a nivel club), o
--   - entrenador_ayudante con can_manage_calendar concedido en el club.
-- Para events con team_id NULL (eventos de club/categoría) solo admin/coord.
create or replace function public.user_can_manage_event(
  p_club_id uuid,
  p_team_id uuid
) returns boolean language sql stable security definer set search_path = public as $$
  select
    public.user_role_in_club(p_club_id) in ('admin_club', 'coordinador')
    or (
      p_team_id is not null
      and (
        public.user_is_staff_of_team(p_team_id)
        and public.user_role_in_club(p_club_id) = 'entrenador_principal'
      )
    )
    or (
      p_team_id is not null
      and public.user_has_capability_in_club(p_club_id, 'can_manage_calendar')
      and public.user_is_staff_of_team(p_team_id)
    );
$$;

create policy events_insert_managers on public.events
  for insert to authenticated
  with check (public.user_can_manage_event(club_id, team_id));

create policy events_update_managers on public.events
  for update to authenticated
  using      (public.user_can_manage_event(club_id, team_id))
  with check (public.user_can_manage_event(club_id, team_id));

create policy events_delete_managers on public.events
  for delete to authenticated
  using      (public.user_can_manage_event(club_id, team_id));
```

**Notas sobre la lección MVCC heredada de NIDO** (memory `feedback_nido_rls_mvcc_gotcha`):

- El helper `user_can_manage_event` consulta `memberships`, `team_staff` y `capabilities`, ninguna de las cuales se modifica en el INSERT/UPDATE a `events`. Por tanto, no hay riesgo MVCC: el `RETURNING *` posterior al INSERT pasa la policy SELECT sin tocar filas mutadas en la misma TX.
- Esto se documentará en un comentario dentro de la migración para que F4 (asistencias, también con RETURNING) replique sin tropezar.

## 5. UI

### 5.1. Decisión D3: componente propio sobre `Intl` + `Date` nativos (sin date-fns)

> **Update durante implementación**: la spec original planteaba `date-fns@^4` como librería base. Al construir el componente vimos que las funciones que íbamos a usar (`startOfWeek`, `addDays`, `eachDayOfInterval`, `format`) se cubren con `Intl.DateTimeFormat` + aritmética simple sobre `Date.UTC`, sin penalización de legibilidad. Se eliminó la dependencia: cero nueva dep en `apps/web`, bundle adicional ≈ 0 KB. Misma decisión "componente propio" (ADR-0006), implementación más ligera. Los helpers viven en `apps/web/src/lib/calendar-utils.ts` y los de TZ-aware en `packages/core/src/events/tz.ts` (reutilizables por el generador de recurrencia).

**Opciones evaluadas**:

| Opción | Peso bundle | Curva | Soporte iPad apaisado | Estética shadcn | Reutilizable F12 |
|---|---|---|---|---|---|
| FullCalendar (open-source plugins) | ~150 KB gz + DOM pesado | Media. API React decente. | Bien con `@fullcalendar/interaction`. | Necesita CSS-override manual. Disonante. | Parcial. |
| react-big-calendar | ~80 KB gz + `moment` o adaptador | Baja. API simple. | Aceptable pero pulido táctil débil. | Idem, requiere override CSS. | Parcial. |
| **Propio sobre date-fns** | ~25 KB gz (date-fns subset) | Alta inicialmente, baja después. | Control total. | 100% Tailwind v4 + Radix. | **Total** (F12 reusa). |

**Elección: propio sobre date-fns**. Razones:

1. **Reutilización F12**: el planificador de sesiones (F12) reutiliza el grid semanal. La inversión ahora ahorra una reescritura completa después.
2. **Bundle**: la PWA es objetivo iPad/Android; cada KB suma. FullCalendar añade más de lo que aporta.
3. **Coherencia**: el repo ya tiene `radix-ui`, `lucide-react`, `tailwindcss@4`, `sonner`, `react-hook-form`, `zod`, `shadcn/ui`. Las primitivas para construir el calendario están todas.
4. **iPad apaisado prioritario**: la app objetivo de partida es iPad. Implementar el grid con CSS Grid + drag/touch nativo da mejor UX que adaptar una librería.
5. **Accesibilidad**: el grid se construye sobre Radix (focus management, ARIA) sin depender de defaults de librerías externas.

**Coste**: ~3–4h extra dentro del rango 6–9h de la fase. Aceptable dado el ahorro futuro.

**Decisión de dependencia**:

```bash
pnpm --filter web add date-fns@^4
```

`date-fns@^4` incluye `@date-fns/tz` nativo desde la v4, sin `date-fns-tz` separado. Se importan solo las funciones usadas: `addDays`, `addWeeks`, `startOfWeek`, `endOfWeek`, `startOfMonth`, `endOfMonth`, `eachDayOfInterval`, `format`, `isSameMonth`, `parseISO`, `toZonedTime`, `fromZonedTime`. Tree-shaken por Next 16.

### 5.3. Vistas — Decisión D4

**Mes**:

- Grid 7 columnas × 5–6 filas. Día actual destacado.
- Cada celda contiene hasta 3 pildoras (`EventPill`) + "N más" si overflow → click abre la vista de día (popover / sheet).
- Color de pildora: por **tipo de evento** principalmente, con borde izquierdo en el **color del equipo** (`teams.color` existente). Eventos de club (sin team) usan color neutro.
- Header: nombre del mes + año. Nav `<` `Hoy` `>`.

**Semana**:

- Grid 7 columnas (lun–dom). Filas en slots de 30 minutos de 07:00 a 23:00 (configurable luego, fijo en F3).
- Eventos posicionados con `grid-row-start` / `grid-row-end` según hora.
- All-day events en una franja superior fija.
- Header: rango `26 may – 1 jun 2026`. Nav `<` `Hoy` `>`.

**Agenda**:

- Lista cronológica agrupada por día.
- Cada día como sección con header sticky.
- Cada evento como card con tipo + hora + título + ubicación + equipo/categoría.
- Vista preferida en móvil portrait. Paginación por 14 días con `Cargar más`.

## 6. Estados, validaciones y errores

### 6.1. Zod schemas (en `packages/core/src/schemas/event.ts`)

```ts
export const recurrenceRuleSchema = z.object({
  freq: z.literal('weekly'),
  interval: z.number().int().min(1).max(4),
  by_weekday: z.array(z.number().int().min(0).max(6)).min(1).max(7),
  // count = número de SEMANAS de la serie (no de hijos generados).
  // Hijos totales = count × by_weekday.length.
  count: z.number().int().min(1).max(52).optional(),
  until: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).optional(),
}).refine(d => (d.count != null) !== (d.until != null), 'count XOR until');

export const eventInputSchema = z.object({
  type: z.enum(['training', 'match', 'tournament', 'friendly', 'other']),
  target: z.discriminatedUnion('kind', [
    z.object({ kind: z.literal('team'),     team_id: z.uuid() }),
    z.object({ kind: z.literal('category'), category_id: z.uuid() }),
    z.object({ kind: z.literal('club') }),
  ]),
  title: z.string().min(1).max(200),
  starts_at: z.iso.datetime(),
  ends_at: z.iso.datetime().nullable(),
  all_day: z.boolean(),
  location_name: z.string().max(160).nullable(),
  location_address: z.string().max(240).nullable(),
  opponent_name: z.string().max(120).nullable(),
  notes: z.string().nullable(),
  recurrence_rule: recurrenceRuleSchema.nullable(),
}).superRefine((d, ctx) => {
  if (d.ends_at && d.ends_at < d.starts_at) {
    ctx.addIssue({ code: 'custom', path: ['ends_at'], message: 'ends_at_before_starts_at' });
  }
  if (d.opponent_name && !['match', 'friendly'].includes(d.type)) {
    ctx.addIssue({ code: 'custom', path: ['opponent_name'], message: 'opponent_only_match_friendly' });
  }
});
```

### 6.2. Generador de recurrencia (en `packages/core/src/events/recurrence.ts`)

Función pura **sin side effects**, sin acceso a clock — testable y determinista:

```ts
export function expandRecurrence(
  parentStartsAt: Date,
  parentEndsAt: Date | null,
  rule: RecurrenceRule,
  tz: string,            // 'Europe/Madrid'
): Array<{ starts_at: Date; ends_at: Date | null }>;
```

- Itera semana a semana respetando `interval` y `by_weekday` en el TZ indicado.
- Para `count`, genera exactamente N instancias **incluyendo** el primer evento (el parent cuenta como instancia 1).
- Para `until`, genera mientras `starts_at <= until` (last-inclusive en el TZ).
- Devuelve los timestamps en UTC (Date instances). El caller persiste como TIMESTAMPTZ.
- Edge cases cubiertos por test:
  - DST (cambio de hora marzo y octubre en Madrid).
  - `count=2` con `by_weekday=[lun, mié]` empezando martes.
  - `until=YYYY-MM-DD` exactamente el último día válido.
  - `interval=2` cada 2 semanas.

### 6.3. Estados de UI

- **Vacío**: sin eventos en el rango → ilustración + CTA "Crear primer evento" (solo si el user puede).
- **Cargando**: skeleton del grid con shapes ghost.
- **Error**: banner con Sonner toast + retry. Sentry captura.
- **Sin permisos**: no se renderiza botón "Nuevo evento". Si una server action falla por RLS, mensaje genérico "No tienes permiso para esta acción" + log Sentry.

### 6.4. Errores específicos

- INSERT con `team_id` + `category_id` ambos no nulos → mensaje "elige equipo, categoría o club" (validación cliente; CHECK la captura en BD por si acaso).
- INSERT con `ends_at < starts_at` → mensaje "la hora de fin debe ser posterior".
- INSERT a evento "club" sin ser admin/coord → server action retorna error tipado.
- Borrado de parent con children: usa cascade (FK on delete cascade) cuando se elige "toda la serie".
