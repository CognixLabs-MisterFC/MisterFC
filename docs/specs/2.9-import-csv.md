# Spec 2.9 — Importación masiva de jugadores vía CSV/Excel

> Subfase del Plan Maestro: **2.9 — Importación masiva de jugadores vía CSV/Excel**.
> Estado: ☑ hecho 2026-05-29
> Autor: Iker Milla
> Fecha de creación: 2026-05-28

---

## 1. Contexto

Un club que arranca con MisterFC trae una plantilla preexistente (típicamente 50-150 jugadores entre varios equipos) en Excel del coordinador, hoja de cálculo del entrenador o exportación de otra herramienta. Dar de alta uno a uno con el form de F2.3 es inviable. Esta subfase introduce el flujo de importación masiva: plantilla descargable, upload, parsing, validación por fila, detección de duplicados, vista previa con feedback visual, confirmación y resumen.

Subfase de **mayor riesgo** dentro de F2 por la cantidad de edge cases (encodings, decimales, fechas en formatos locales, columnas en orden distinto, duplicados parciales). Por eso va en su propio lote D.

## 2. Objetivos

- Plantilla descargable (XLSX) en `/plantilla-import/template.xlsx` o equivalente público, con columnas en orden fijo, ejemplo de datos en la primera fila y hoja de instrucciones.
- Página `/plantilla/importar` accesible a admin_club, coordinador, entrenador_principal y ayudante con `can_manage_squad`.
- Carga de fichero `.csv` o `.xlsx`. Tamaños hasta 5 MB / 500 filas en esta primera versión.
- Parsing en cliente (papaparse + SheetJS/xlsx, ambos lightweight). Validación por fila con los mismos schemas Zod que F2.3.
- Detección de duplicados frente a `players` existentes en el club: clave compuesta `(lower(first_name), lower(last_name), date_of_birth, club_id)`. Match exacto = "ya existe, se omite". Sin merge automático. (Filas sin `date_of_birth` no llegan al dedup: la columna es obligatoria en `players` y en la plantilla, así que esas filas quedan en rojo desde validación — ver §6 "Comportamiento sin date_of_birth").
- Vista previa con tabla coloreada:
  - **Verde**: fila válida, lista para insertar.
  - **Amarillo**: fila válida pero duplicado en BD → se omite.
  - **Rojo**: fila inválida (campos obligatorios faltan o formato incorrecto) → no se inserta. Muestra el primer error de validación por fila.
- Selección de equipo destino opcional: si el user elige uno, todos los jugadores válidos importados se asignan a ese `team_id` vía `team_members`. Si no, los jugadores quedan en el club sin equipo (pendientes de asignación manual).
- Confirmación → server action `importPlayers(rows[], teamId | null)` que valida en server (Zod por fila + check de pertenencia al club) y hace INSERT batch.
- Resumen final: "X jugadores creados, Y omitidos por duplicado, Z fallidos". Descarga del informe en CSV con el detalle por fila para auditoría.

## 3. Fuera de alcance

- **Mapping de columnas configurable**: la plantilla tiene columnas en orden fijo. Si el user trae un Excel con columnas distintas, debe rellenar nuestra plantilla. Cargar mapping interactivo añade complejidad y se posterga.
- **Importar `medical_notes`**: dato sensible, se carga manualmente desde la ficha del jugador. Si la plantilla lo trae, se ignora con aviso.
- **Importar fotos**: no. F2.2 sube foto desde la ficha individual.
- **Importar player_accounts**: el alta de tutores es manual en F2.4.
- **Update masivo** (modificar jugadores existentes por CSV): solo INSERT en esta subfase.
- **Asignación a múltiples equipos** por jugador: el formulario solo permite un team destino. Asignación a más equipos vía F2.5 (histórico) o ficha individual.
- **Multi-club**: la importación siempre va al club activo. Cross-club no aplica.

## 4. Modelo de datos afectado

**Cero SQL nuevo**. Reusa `players` y `team_members`.

INSERT batch:

- Por fila válida no duplicada: `INSERT INTO players (...)` con `club_id` = club activo del user.
- Si `teamId` está seleccionado: `INSERT INTO team_members (player_id, team_id, joined_at)` por cada player creado.

Transaccionalidad: cada fila se procesa de forma independiente (un error en la fila N no aborta las anteriores). Si una fila falla en server (race condition contra un duplicado creado en otra pestaña), se cuenta en "fallidos" con motivo "duplicate_constraint" o similar.

Plantilla:

```
apps/web/public/import-templates/players-template.xlsx        ← Para descarga directa
apps/web/public/import-templates/players-template.csv         ← Alternativa CSV
```

### Wizard (4 pasos)

1. **Subir archivo**: input file (`.csv,.xlsx`) + botón "Descargar plantilla". Validación de tamaño y MIME en cliente.
2. **Vista previa**: tabla con todas las filas, coloreadas. Selector "Asignar al equipo" (dropdown de equipos del club, opcional). Contadores arriba: "X válidas, Y duplicadas, Z inválidas".
3. **Confirmación**: botón "Importar X jugadores" (solo cuenta verdes). Modal de confirmación.
4. **Resultado**: resumen + tabla con el resultado por fila + botón "Descargar informe CSV".

### Plantilla de columnas

```
first_name (obligatorio)
last_name  (obligatorio)
date_of_birth (obligatorio, formato ISO YYYY-MM-DD o DD/MM/YYYY)
dorsal (opcional, 1-99)
position_main (opcional, goalkeeper/defender/midfielder/forward)
positions_secondary (opcional, separadas por |, ej: "defender|midfielder")
foot (opcional, right/left/both)
height_cm (opcional, 50-250)
weight_kg (opcional, 10.00-200.00 con coma o punto decimal)
origin (opcional, texto libre <= 120 chars)
```

`medical_notes` y cualquier columna extra se ignora silenciosamente con aviso al user.

## 6. Estados, validaciones y errores

### Esquemas Zod en `packages/core/src/schemas/import.ts`

- `playerImportRowSchema`: refleja la plantilla con coerciones explícitas:
  - `date_of_birth`: acepta `YYYY-MM-DD`, `DD/MM/YYYY`, `DD-MM-YYYY`. Normaliza a ISO.
  - `dorsal`: coerce a número, valida rango.
  - `position_main`: trim + lowercase, valida enum.
  - `positions_secondary`: split por `|`, trim, valida cada uno contra el enum.
  - `foot`: trim + lowercase, valida enum.
  - `height_cm`, `weight_kg`: coerce numérico, valida rango.
- `playerImportPayloadSchema`: array de `playerImportRowSchema` con length 1-500.

### Estados de UI

- **Sin archivo**: paso 1 visible, botón "Continuar" deshabilitado.
- **Archivo subido**: parsing en cliente. Spinner. Si falla parsing (encoding, no es CSV/XLSX), mensaje de error y vuelve a paso 1.
- **Parsing OK**: paso 2 con tabla coloreada. Contadores en vivo.
- **Confirmación en flight**: spinner, deshabilitar interacción, mostrar progreso por chunks (si > 100 filas, dividir en chunks de 100 al enviar al server).
- **Resultado**: paso 4 con números y CTA a `/plantilla` para ver los importados.

### Errores esperados

- **Encoding raro**: muchos Excel españoles guardan en `windows-1252`. papaparse intenta detectar; si falla, mostrar mensaje sugiriendo "guardar como CSV UTF-8".
- **Fecha en formato local (DD/MM/YYYY con barras)**: aceptada por el schema, normalizada.
- **Dorsales duplicados en el archivo** (dos filas con dorsal 10): aceptado en BD porque el dorsal en `players` es opcional y sin UNIQUE. Aviso visual en preview.
- **Posición secundaria mal escrita**: fila inválida, motivo "position invalid".
- **Race condition**: dos imports simultáneos del mismo CSV → la segunda detecta duplicados en server y cuenta en omitidos.

### Comportamiento sin date_of_birth

`date_of_birth` es **obligatorio** en `players` (NOT NULL en BD) y en la plantilla. Por tanto:

- Fila sin `date_of_birth` → schema falla → **fila roja**, motivo `date_of_birth_required`. **No se importa**.
- No llega al dedup, no entra en BD.
- Razón del diseño: sin DOB no se puede diferenciar entre dos jugadores con mismo nombre+apellido (escenario habitual: dos hermanos), y la app necesita la edad para asignar categorías. Importar sin DOB nos dejaría con datos a medias que luego hay que completar a mano uno a uno, peor que no importarlos.
- Si el club no tiene la DOB de algún jugador, el flujo correcto es: importar el resto, dar de alta esos casos puntuales desde el form individual de F2.3 cuando alguien recuerde la fecha. La plantilla deja claro este requisito.

### Errores tipados de la action

```ts
type ImportResult = {
  created: number;
  skipped_duplicates: number;
  failed: number;
  details: Array<{
    row_index: number;
    status: 'created' | 'skipped' | 'failed';
    reason?: string;
    player_id?: string;
  }>;
};
```

## 7. Tests

- **Unit (Vitest) — añadidos en este lote D**. Es el primer Vitest del repo. Setup mínimo (`vitest` + `@vitejs/plugin-react` si lo pide algún test JSX; aquí basta TS puro) en `apps/web` o en `packages/core`, lo que encaje (preferencia: lógica pura en `packages/core` con tests ahí). Cobertura objetivo:
  - **Parsing** (`parse-file`):
    - CSV UTF-8 con cabeceras correctas → array de rows con campos snake_case.
    - CSV `windows-1252` con `áéíóúñ` → caracteres legibles tras decode.
    - XLSX con varias hojas → solo lee la primera (o la nombrada "jugadores" si existe).
    - Archivo vacío / sin cabeceras → error tipado, no crash.
    - Columnas extra inesperadas → se ignoran, no rompen.
  - **Validación por fila** (`playerImportRowSchema`):
    - Fechas en formato `YYYY-MM-DD`, `DD/MM/YYYY`, `DD-MM-YYYY` → normalizadas a ISO.
    - Fecha sin `date_of_birth` → falla con código `date_of_birth_required`.
    - `dorsal` fuera de rango (`0`, `100`, `-1`, `abc`) → falla.
    - `position_main` mal escrita (`midfielder ` con espacio, `MIDFIELDER`) → trim + lowercase, válida.
    - `position_main` inválida (`striker`) → falla con código `position_invalid`.
    - `positions_secondary` con `|` → split correcto; con uno inválido falla.
    - `weight_kg` con coma decimal (`72,5`) y punto (`72.5`) → ambos parsean a `72.5`.
    - `height_cm` fuera de rango → falla.
  - **Detección de duplicados** (`detect-duplicates`):
    - Mismas filas (idéntico nombre+apellido+DOB) → la primera marca created, la segunda skipped.
    - Mismo nombre+apellido pero DOB distinta → ambas created (caso hermanos).
    - Capitalización distinta (`pepe gomez` vs `Pepe Gómez`) → considera duplicado solo si normalizamos por `lower()` y trim; tildes NO se normalizan (asumimos input limpio). Documentar este caso en el test.
    - Fila local sin DOB → no llega al dedup, ya filtrada por validación.

  Estos tests viven en `packages/core/src/import/__tests__/` y se ejecutan con `pnpm --filter @misterfc/core test`. Si el setup de Vitest provoca fricción en el monorepo (ej. transformers), reportar y aplicar fix en su propio commit dentro del lote.
- **RLS (pgTAP)** — añadir `supabase/tests/rls_players_import_batch.sql`:
  - Admin/coord/principal pueden INSERT batch en su club.
  - Ayudante con `can_manage_squad=true` puede INSERT.
  - Ayudante sin `can_manage_squad` NO puede.
  - Jugador NO puede.
  - User de otro club NO puede insertar con `club_id` del club ajeno (RLS lo rechaza por fila).
- **E2E (Playwright)** — pospuesto a F14.
- **Smoke manual** documentado en el PR (ver §10).

## 8. Notas de implementación

- **Parsers**:
  - CSV: `papaparse` (~17 KB gz, sin deps). Modo `{ header: true, skipEmptyLines: true, transformHeader: snake }`.
  - XLSX: `xlsx` o `xlsx-populate`. SheetJS `xlsx` es la opción canónica pero pesa (~280 KB gz). Alternativa: `read-excel-file` (~30 KB gz, sin escritura). **Recomendado**: `read-excel-file` para read-only y bundle pequeño. Decidir al implementar; si se elige `xlsx`, lazy-load detrás de `await import()` para no impactar el bundle de las páginas no-import.
- **Generación de la plantilla**: pre-generar el XLSX/CSV manualmente con datos de ejemplo y commitearlos en `apps/web/public/import-templates/`. No generamos dinámicamente para no añadir SheetJS al bundle servidor.
- **Server action batch**: recibe `rows: PlayerImportRow[]` y `teamId: string | null`. Valida cada fila con Zod (defense in depth — el cliente ya validó pero el server es la autoridad). Loop con INSERTs individuales para que un error en fila N no aborte las otras (no envolver en BEGIN/COMMIT manual; cada `await supabase.from('players').insert(...)` es su propia transacción).
  - Alternativa más rápida: `insert([...])` array batch. Pero ante el primer error, falla todas. Para esta versión preferimos loop individual y aceptamos N round-trips.
  - **Si performance se hace problema** (importar 500 jugadores se va a > 10s): mover a Edge Function que use service role + transacción explícita. **No en esta subfase**, anotado para F11.
- **Duplicate detection en server**: antes del INSERT, query `select id from players where club_id = $1 and lower(first_name) = lower($2) and lower(last_name) = lower($3) and date_of_birth = $4`. Si existe, skip. Race condition mínima (dos imports simultáneos): el segundo INSERT sigue adelante porque no hay UNIQUE constraint; lo aceptamos como caso edge (improbable, daño bajo). Si se hace común, añadir UNIQUE parcial.
- **Asignación a equipo**: si `teamId` viene, tras cada INSERT exitoso de player, hacer INSERT en `team_members(player_id, team_id, joined_at=current_date)`. Si esa segunda INSERT falla (improbable), el player queda creado sin equipo y el detalle lo refleja.
- **Informe descargable**: generar el CSV en cliente con los datos del `ImportResult` usando una util simple (sin papaparse para encode — basta string join + escape). No persistir el informe en BD.
- **i18n**: `import.*` namespace en `messages/{es,en,va}.json`. Mensajes de error de validación específicos por columna.
