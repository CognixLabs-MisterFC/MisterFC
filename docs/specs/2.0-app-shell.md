# Spec 2.0 — App shell + navegación role-aware + perfil propio

> Subfase del Plan Maestro: **2.0 — App shell + navegación role-aware + fix logout + perfil propio**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-28

---

## 1. Contexto

Fase 1 cerrada. La home (`/[locale]/page.tsx`) muestra hoy una lista de clubes y un botón de logout, sin navegación real. Cualquier módulo nuevo de Fase 2 (categorías, equipos, jugadores, staff) necesita una capa de chrome común: sidebar/drawer, breadcrumbs, indicador de club activo, logout siempre accesible y página de perfil propio. Esta subfase es **foundational**: bloquea visualmente al resto de F2 y arregla el known-issue de F1 ("logout ausente en /onboarding").

Decisión implícita del plan: el rol del user en el **club activo** determina qué entradas de menú ve. Si pertenece a varios clubes, necesita selector de club activo.

## 2. Objetivos

- Layout autenticado compartido por todas las páginas dentro de un club: sidebar fijo en desktop (≥ 1024 px) y drawer colapsable en móvil/tablet, header con club activo + perfil + logout.
- Menú dinámico por rol:
  - **admin_club**, **coordinador**: Inicio · Categorías · Equipos · Plantilla del club · Staff · Invitaciones · Mi perfil.
  - **entrenador_principal**, **entrenador_ayudante**: Inicio · Mis equipos · Mi plantilla · Mi perfil.
  - **jugador**: Inicio · Mi ficha · Mi perfil.
- Selector de club activo cuando el user tiene ≥ 2 memberships. El club activo se persiste en cookie `active_club_id` (httpOnly, SameSite=Lax, sin firma — basta con que el servidor valide que la cookie referencia un club al que el user pertenece).
- Botón "Cerrar sesión" visible en **todas** las páginas autenticadas, incluida `/onboarding` (resuelve el known-issue).
- Página `/perfil`: el user edita `full_name`, `date_of_birth`, `locale`, y sube/cambia su `avatar_url` (bucket `profile-avatars`, RLS por user).
- Navegación tipada vía `@/i18n/navigation` (`Link`, `useRouter` exportados por next-intl). Sin strings de ruta sueltos.

## 3. Fuera de alcance

- **shadcn/ui**: contemplado en ADR-0000 y compatible con Tailwind v4 + Next.js 16. **Se introduce en el lote A** y se usa para los componentes nuevos de F2 (forms, dropdowns, switches, dialogs, sheet para drawer móvil, table para preview). Si al ejecutar `pnpm dlx shadcn@latest init` apareciera un bloqueo técnico real con la combinación T4 + Next 16, se reporta y se cae a Tailwind manual con ADR-0005 que documente el motivo. **No es opt-out por preferencia.**
- Slugs por club en la URL (`/c/[slug]/...`). El club activo va por cookie + middleware. El plan multi-tenant URL-aware llega cuando el primer cliente lo pida.
- Cambio de email del user (lo gestiona Supabase Auth, lo abordamos en F11 o cuando aparezca necesidad).
- Notificaciones in-app o badges en el sidebar (llegan en F5).
- Temas claro/oscuro. Modo oscuro de salida; preferencia configurable en futuro.

## 4. Modelo de datos afectado

**Cero SQL nuevo en esta subfase**. Reusa `profiles` (avatar_url, full_name, date_of_birth, locale) y `memberships` (multi-club).

Storage nuevo:

- Bucket **`profile-avatars`** (**privado**, acceso vía signed URLs).
  - Decisión: privado por **coherencia con el contexto RGPD del producto** (datos de menores en F2.2 onwards) y para evitar leakage incidental de avatares vía URLs CDN cacheadas indefinidamente. El avatar de un user no es PII grave en sí mismo, pero un bucket público asienta un patrón laxo que se replica con menos rigor; mejor cerrar desde el principio.
  - Naming convention: `<profile_id>/<uuid>.<ext>` (carpeta por user para policies simples y para que el path solo sea adivinable conociendo el user_id, que ya está restringido).
  - Tipos permitidos: `image/jpeg`, `image/png`, `image/webp`. Tamaño máximo: 2 MB (validación cliente + check server-side antes de generar URL).
  - Lectura: signed URLs con TTL de 1 hora generadas en server (`supabase.storage.from('profile-avatars').createSignedUrl(path, 3600)`). En `profiles` se guarda el **path** (`profile_id/uuid.webp`), no la URL firmada (que expira). El render renueva la firma a demanda.
  - Policies:
    - SELECT: `to authenticated using (bucket_id = 'profile-avatars')` — cualquier user autenticado puede generar signed URL de cualquier avatar del bucket. Esto es aceptable porque el path no es enumerable (UUIDs) y solo se exponen los paths via consultas a `profiles` que la RLS de `profiles` ya filtra (clubmates).
    - INSERT/UPDATE/DELETE: solo en propia carpeta.

Migración:

```sql
-- supabase/migrations/<ts>_storage_profile_avatars.sql
insert into storage.buckets (id, name, public)
values ('profile-avatars', 'profile-avatars', false)  -- privado
on conflict (id) do nothing;

create policy "profile_avatars_select_authenticated" on storage.objects
  for select to authenticated
  using (bucket_id = 'profile-avatars');

create policy "profile_avatars_write_own" on storage.objects
  for insert to authenticated
  with check (
    bucket_id = 'profile-avatars'
    and (storage.foldername(name))[1] = auth.uid()::text
  );

create policy "profile_avatars_update_own" on storage.objects
  for update to authenticated
  using (
    bucket_id = 'profile-avatars'
    and (storage.foldername(name))[1] = auth.uid()::text
  );

create policy "profile_avatars_delete_own" on storage.objects
  for delete to authenticated
  using (
    bucket_id = 'profile-avatars'
    and (storage.foldername(name))[1] = auth.uid()::text
  );
```

**`profiles.avatar_url`**: hoy es la URL completa con check `~ '^https?://'`. Con signed URLs no podemos guardar URL completa (expira). Plan:
- Renombrar conceptualmente: `avatar_url` pasa a guardar el **path** del objeto (`<profile_id>/<uuid>.webp`).
- Relajar el CHECK o, mejor, **migración aparte** que cambie la columna de `avatar_url text` a `avatar_path text` con CHECK relajado a `char_length(...) <= 200 and avatar_path not like 'http%'`. Decidir al implementar: si renombrar provoca cascada de cambios (componentes que asumen URL), mantener `avatar_url` como nombre pero contenido = path, con CHECK actualizado. Documentar la decisión inline.

### Nota crítica para el lote B (F2.2)

El bucket **`player-photos` debe ser privado y NO replicar el patrón "público" descartado aquí**. Razones:

1. Son fotos de **menores** — RGPD lo trata como dato especialmente sensible.
2. La visibilidad debe estar restringida a **miembros del club del jugador**, no a "cualquier authenticated" como `profile-avatars`. Esto implica que la SELECT policy debe hacer un lookup tipo `exists (select 1 from players p where p.id::text = (storage.foldername(name))[1] and user_role_in_club(p.club_id) is not null)` o equivalente, con naming convention `<player_id>/<uuid>.<ext>`.
3. Acceso vía **signed URLs cortas** (TTL 5-15 min) generadas server-side tras verificar pertenencia. NO usar URL pública ni un TTL largo cacheable.
4. Considerar helper SQL nuevo `public.user_can_see_player(p_player_id uuid) returns boolean` que centralice la lógica (member del club del player + check de medical-notes-visibility si aplica). La RLS de storage lo reutiliza.

Esta nota es vinculante para spec 2.2 — no se mergea el lote B sin estas restricciones.

## 5. UI

### Estructura de rutas

```
apps/web/src/app/[locale]/
├── (authenticated)/          (route group, no afecta URL)
│   ├── layout.tsx            ← Shell: sidebar + header + outlet
│   ├── page.tsx              ← Dashboard mínimo "Bienvenido a {club}"
│   ├── perfil/page.tsx
│   ├── perfil/perfil-form.tsx
│   ├── perfil/actions.ts
│   └── (más rutas en lotes B/C/D)
├── onboarding/page.tsx       ← FUERA del group: usuario sin club, header mínimo con logout
├── signin/, signup/, ...     ← Sin layout autenticado
```

El **route group `(authenticated)`** envuelve todo lo que requiere `getCurrentUser` + `getCurrentUserClubs.length > 0`. Su `layout.tsx` redirige a `/signin` o `/onboarding` cuando falta sesión o club. La página `/onboarding` queda fuera del group para no caer en bucle, pero recibe un header minimalista con logout (componente compartido).

### Componentes nuevos en `apps/web/src/components/shell/`

- `<AppShell>` (server): orquesta sidebar + header + main. Recibe `user`, `activeClub`, `role`, `memberships`.
- `<Sidebar>` (server): renderiza la lista de entradas según `role`. Highlight de la entrada activa vía `usePathname` (client component pequeño wrapper).
- `<MobileDrawer>` (client): controla open/close en móvil. Trigger en header.
- `<ActiveClubSwitcher>` (client): dropdown si `memberships.length > 1`. Server action `setActiveClub(clubId)` que valida pertenencia y actualiza cookie + revalida.
- `<UserMenu>` (client): avatar + dropdown con "Mi perfil", "Cerrar sesión".
- `<LogoutButton>` (server-action form): reutilizable, plain `<form action>`. Usado por header y por `onboarding`.

### Página `/perfil`

Form con tres bloques:

1. Avatar: vista actual + `<input type="file">` + preview + acción "Guardar foto" (sube al bucket y UPDATE `profiles.avatar_url`).
2. Datos: `full_name`, `date_of_birth` (date picker nativo), `locale` (select es/en/va).
3. Cuenta: email (read-only), enlace "Cambiar contraseña" → `/forgot-password` ya existente.

Patrón `useActionState` + Server Action `updateProfile(formData)` retornando `FormState`. Validación con Zod.

### Selector de club activo

Si `memberships.length === 1`, el header pinta el nombre del club como texto plano. Si `> 1`, dropdown que invoca server action `setActiveClub(clubId)`. El menú del sidebar se re-renderiza con el rol del nuevo club.

La cookie `active_club_id` no es de seguridad — solo UX. Toda RLS sigue siendo por `user_role_in_club(club_id)`. Si la cookie apunta a un club al que el user no pertenece (manipulada o stale), el layout fallback: usa la primera membership y reescribe la cookie.

## 6. Estados, validaciones y errores

### Esquemas Zod nuevos en `packages/core/src/schemas/profile.ts`

- `updateProfileSchema`: `full_name` (2-120, trim), `date_of_birth` (date ISO, no futura, >= 1900, opcional), `locale` (enum es/en/va).
- `uploadAvatarSchema`: `mimeType` ∈ {image/jpeg, image/png, image/webp}, `size` ≤ 2 MB.

### Estados de UI

- **Sin sesión**: middleware redirige a `/signin` (ya implementado).
- **Sesión + sin club**: `(authenticated)/layout` redirige a `/onboarding`.
- **Cookie `active_club_id` apunta a club no perteneciente**: layout escoge la primera membership y reescribe cookie. Sin error visible.
- **Avatar upload > 2 MB**: error UX cliente, no llega al server.
- **Mime no permitido**: idem.
- **Network error en upload**: pinta `t('perfil.errors.avatar_upload_failed')`, no rompe form.

### Edge cases

- User cambia su `locale` en `/perfil`: la cookie de next-intl (`NEXT_LOCALE` si existe) y el `profiles.locale` deben quedar coherentes. La action redirige a `/{newLocale}/perfil` tras guardar.
- User borra su avatar (futuro, no en esta subfase): no hay UI todavía. La política DELETE existe pero no se invoca.
- User en `/onboarding` pulsa logout: cierra sesión y redirige a `/signin`. Sin redirección a `/` (no tiene club).

## 8. Notas de implementación

- Toda i18n nueva en `messages/{es,en,va}.json` bajo namespace `shell.*` y `perfil.*`. `es` lleva el texto canónico; `en` y `va` se pueden estub con el mismo texto si no hay traducción todavía (queda anotado para F14).
- El `(authenticated)/layout.tsx` lee user + memberships una sola vez y los pasa por props a `<AppShell>`. Evitar re-fetch en cada Server Component hijo: si se necesita el club activo en un hijo profundo, usar un Server Component contextual o leer la cookie directamente.
- `setActiveClub` valida que `clubId` pertenece a los memberships del user vía `user_role_in_club(clubId) is not null` (read-side, no escritura). Si pasa, hace `cookies().set('active_club_id', clubId, { httpOnly: true, sameSite: 'lax', path: '/', maxAge: 60*60*24*365 })` y `revalidatePath('/', 'layout')`.
- El cambio de locale via `/perfil` requiere `cookies().set('NEXT_LOCALE', newLocale, ...)` además del UPDATE en `profiles`. Confirmar con next-intl docs el nombre exacto de la cookie.
- Avatar upload: usar `supabase.storage.from('profile-avatars').upload(...)` desde el cliente con el client SSR público, no service role. La RLS de storage hace el resto.
- Lookups de imagen: `supabase.storage.from('profile-avatars').getPublicUrl(path)` devuelve URL CDN sin firma. Guardar esa URL en `profiles.avatar_url`.
