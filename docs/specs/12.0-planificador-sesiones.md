# Spec 12.0 — Planificador de sesiones

> Fase del Plan Maestro: **F12 — Planificador de sesiones con plantillas microciclo**.
> Estado: ☐ pendiente

## 1. Contexto

F11 dejó la **biblioteca de ejercicios** (modelo `exercises`, listado/ficha, contrato del diagrama, editor `<PitchBoard>`/`<DiagramView>`) y el **ciclo de metodología del club** como pieza reutilizable. F12 construye **encima**: el entrenador monta una **sesión de entrenamiento** arrastrando ejercicios de la biblioteca dentro de una estructura de **bloques**, con objetivos y meso/microciclo, y la publica al equipo. La estructura objetivo se derivó de la **hoja de sesión real** del cliente (Infantil B, 11/06/2026), capturada en [spec 11.0 §13.2](11.0-biblioteca-ejercicios.md): cabecera (objetivos físico/táctico/técnico + meso/microciclo + tiempo total) + bloques (calentamiento, complementaria, principal×2, vuelta a la calma); cada tarea con su **duración/series DEL DÍA**.

Esta spec **reúsa** todo lo que ya existe y solo añade: el modelo de sesión, el editor, la vista microciclo, la publicación al equipo, el PDF y las plantillas. **F12 NO rediseña** el ciclo de metodología ni la biblioteca.

**Dependencias previas cubiertas**: F11 cerrada (exercises + diagrama + editor), F2.10 (patrón listado), F1.4 (capability `can_create_sessions`, ya sembrada) + F1.7 (helpers RLS), F6 (`FieldMarkings`), F9.B (infra PDF).

## 2. Objetivos

- El entrenador arma una sesión en **bloques** (esqueleto estándar sembrado) arrastrando ejercicios de la biblioteca, con **override de duración/series del día** por tarea.
- Cabecera con objetivos (reusa los vocabularios de F11) + meso/microciclo + tiempo total.
- **Vista microciclo** (semana) y listado de sesiones.
- **Publicar al equipo**: jugadores y familias ven la sesión read-only por adelantado.
- **PDF** de la sesión con el formato de la hoja del anexo.
- **Plantillas**: guardar una sesión como plantilla y crear sesiones desde ella (clona bloques + ejercicios a una fecha nueva).

## 3. Decisiones cerradas (D1–D8)

| # | Decisión |
|---|---|
| **D1** | **Bloques**: catálogo **FIJO** en core (constante estilo `CATEGORY_KINDS`): `calentamiento`, `complementaria`, `principal`, `vuelta_a_la_calma`. Al crear sesión se **siembra** el esqueleto estándar: **5 bloques en orden** = calentamiento, complementaria, principal, principal, vuelta_a_la_calma. Un bloque puede quedar **vacío**. Reordenar + `title`/`notes` opcionales **sí**; añadir/quitar **tipos** = backlog. (Fijo en v1; configurable por club = futuro F17.) |
| **D2** | **Sesiones = creación directa.** **NADA** de ciclo `draft→proposed→published`. Trigger simple (owner forzado a `auth.uid()`, owner/club inmutables, `updated_at`), **sin** máquina de estados. |
| **D3** | **`visibility 'staff'\|'team'`** (default `staff`, patrón `lineups`). En `'team'` → **jugadores Y familias** del `team_id` la ven **read-only y por adelantado** (`/mi-equipo`). |
| **D4** | **`event_id`** (FK `events`, tipo training) **NULLABLE**. Si está, la sesión aparece en calendario / mi-equipo. No obligatorio. |
| **D5** | **Plantillas** (12.6): "Guardar sesión como plantilla" + "Crear sesión desde plantilla" (clona bloques + ejercicios a fecha nueva). Del entrenador (owner), club-scoped, **SIN ciclo de aprobación**. Una plantilla = **una sesión (un día)**; microciclo entero = backlog. Implementación: **`sessions.is_template`** (sin fecha) — misma tabla, no tabla aparte (lo más limpio: el editor, la RLS y el clonado se reutilizan). |
| **D6** | **PDF** (12.5) = hoja del anexo: cabecera + bloques + tareas (texto + duración/series). **Diagramas en PDF = follow-up** (`@react-pdf` no pinta `<DiagramView>` SVG directo → requiere rasterizar). Reúsa infra 9.B. |
| **D7** | **meso/microciclo** = **campos de cabecera** (texto/número) en v1. Plan de temporada estructurado (macro→meso→micro) = backlog. |
| **D8** | **Cabecera** reúsa los vocabularios `tactical_objectives`/`technical_objectives` de exercises + `objective_physical` libre. Estos + la **categoría del equipo** alimentan los **filtros del picker** (reúsa filtros 11.3 + picker 11B.1): el entrenador filtra por categoría + objetivos y elige ejercicios por bloque. |

## 4. Modelo de datos

Calca el patrón de `exercises` (F11.1): `owner_profile_id` forzado por trigger, `club_id`, inmutabilidad de owner/club, `club_id` denormalizado en hijas para RLS sin joins, índices por `(club_id, …)`. **Migración nueva**; sin tocar tablas existentes (salvo, si D4, lectura de `events`).

```sql
-- sessions — cabecera de la sesión (o plantilla si is_template).
create table public.sessions (
  id                  uuid primary key default gen_random_uuid(),
  owner_profile_id    uuid not null references public.profiles(id) on delete cascade,
  club_id             uuid not null references public.clubs(id)    on delete cascade,
  team_id             uuid references public.teams(id) on delete set null,   -- equipo destino
  is_template         boolean not null default false,                        -- D5
  session_date        date,                 -- NULL si is_template
  title               text check (title is null or char_length(title) between 1 and 120),
  objective_physical  text,                 -- D8 (libre)
  tactical_objectives  text[] not null default '{}',   -- D8: mismo vocabulario que exercises
  technical_objectives text[] not null default '{}',   -- D8
  mesocycle           text,                 -- D7 (libre)
  microcycle          text,                 -- D7 (libre)
  total_minutes       smallint check (total_minutes is null or total_minutes >= 0),
  event_id            uuid references public.events(id) on delete set null,  -- D4 (training, nullable)
  visibility          text not null default 'staff' check (visibility in ('staff','team')), -- D3
  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now()
  -- CHECK: una sesión real tiene fecha; una plantilla no.
  -- check ((is_template and session_date is null) or (not is_template and session_date is not null))
);

-- session_blocks — bloques ordenados (esqueleto sembrado al crear).
create table public.session_blocks (
  id          uuid primary key default gen_random_uuid(),
  session_id  uuid not null references public.sessions(id) on delete cascade,
  club_id     uuid not null references public.clubs(id) on delete cascade,   -- denorm para RLS
  block_type  text not null check (block_type in
                ('calentamiento','complementaria','principal','vuelta_a_la_calma')), -- D1
  title       text,
  notes       text,
  order_idx   smallint not null,
  unique (session_id, order_idx)
);

-- session_block_exercises — join con OVERRIDE DEL DÍA (no va en el ejercicio).
create table public.session_block_exercises (
  id           uuid primary key default gen_random_uuid(),
  block_id     uuid not null references public.session_blocks(id) on delete cascade,
  session_id   uuid not null references public.sessions(id) on delete cascade, -- denorm
  club_id      uuid not null references public.clubs(id) on delete cascade,    -- denorm para RLS
  exercise_id  uuid not null references public.exercises(id) on delete restrict,
  order_idx    smallint not null,
  duration_min smallint check (duration_min is null or duration_min >= 0),  -- "18 min"
  series       text,    -- "2 x 8'" (override del día — spec 11.0 §339)
  notes        text,    -- ajuste del día
  unique (block_id, order_idx)
);
```

**Vocabularios** (D8): `tactical_objectives`/`technical_objectives` deben **validar contra los mismos enums que `exercises`** (constantes de `@misterfc/core`). Para no duplicar el `CHECK ... <@ array[...]` literal, valida en la **capa de app** (zod con los enums de core) + un `CHECK` ligero opcional; el set canónico vive en core (mantener en sync, igual que en exercises). El catálogo de **tipos de bloque** se añade a core como `SESSION_BLOCK_TYPES` + el esqueleto `DEFAULT_SESSION_SKELETON` (orden sembrado), con Vitest.

### 4.1 Helper de autoridad

```sql
-- Quién puede crear/editar sesiones del club (calca user_can_create_exercises):
-- admin/coord, principal de algún team, o staff con capability can_create_sessions.
create or replace function public.user_can_create_sessions(p_club_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select
    public.user_role_in_club(p_club_id) in ('admin_club','coordinador')
    or public.user_has_capability_in_club(p_club_id, 'can_create_sessions')
    or exists ( /* principal de algún team del club, calcado de exercises */ );
$$;
```

### 4.2 Trigger (simple — D2)

`sessions_validate` (BEFORE INSERT/UPDATE): en INSERT fuerza `owner_profile_id := auth.uid()`; en UPDATE prohíbe cambiar `owner_profile_id`/`club_id` y setea `updated_at`. **Sin** transiciones de estado (no hay estado). Las hijas heredan `club_id` por el trigger o se pasa desde la app (denormalizado, validado contra el padre).

### 4.3 RLS

- **SELECT**: staff del club (`admin_club/coordinador/entrenador_principal/entrenador_ayudante`) ve todo; **jugador/familia** ve la sesión **solo si `visibility='team'` y pertenece al `team_id`** (reusa el patrón de pertenencia de F4/F5: `team_members` + `player_accounts`). Plantillas (`is_template`) → solo staff (nunca jugador). Las hijas (`session_blocks`, `session_block_exercises`) heredan la visibilidad vía `club_id` + EXISTS sobre la sesión padre (o se replica el predicado con el `club_id` denorm + comprobación de visibilidad/equipo).
- **INSERT/UPDATE/DELETE**: `owner_profile_id = auth.uid()` ∪ `admin_club`, con `user_can_create_sessions(club_id)`. Cascadas por `session_id`/`block_id` (las hijas no necesitan DELETE propio si se borran por cascada, pero sí policies INSERT/UPDATE para el editor).
- **pgTAP**: matriz tabla×rol×operación (calca `rls_exercises.sql`): staff CRUD, ayudante con/sin capability, jugador del team ve `team` y no ve `staff`, jugador de otro team no ve nada, familia vía `player_accounts`, plantillas invisibles a jugador. **Verificación contra el remoto** (deuda **F15.8**: pgTAP no corre en CI).

## 5. UI

- **`/sesiones`** (autenticada, staff): listado (reusa patrón F2.10: filtros equipo/categoría/fecha + `.range()`). Tarjeta en el hub **Entrenamientos** (junto a Ejercicios/Asistencia/Pizarra).
- **`/sesiones/nueva`** y **`/sesiones/[id]/editar`** (gating `can_create_sessions`): editor con cabecera (objetivos con los selectores de F11 + meso/micro + fecha/equipo) + **bloques sembrados** (esqueleto D1) reordenables + por bloque, **añadir ejercicio** vía **picker filtrado** (categoría del equipo + objetivos, reusa filtros 11.3 + picker 11B.1) con **override duración/series/notas del día** + reordenar tareas.
- **`/sesiones/[id]`**: ficha read-only de la sesión (cabecera + bloques + tareas con override + enlace a la ficha de cada ejercicio / su diagrama vía `<DiagramView>`), botón **PDF** y, si staff, **editar/publicar/guardar como plantilla**.
- **Vista microciclo (semana)** (12.3): sesiones del equipo en la semana, derivada de `session_date` (o `event_id`).
- **Jugador/familia** (12.4): en `/mi-equipo`, las sesiones `visibility='team'` del equipo, read-only.
- **i18n** es/en/va de todo lo nuevo (tipos de bloque, etiquetas del editor, PDF).

## 6. PDF (D6)

Reúsa `apps/web/src/lib/pdf/shared.tsx` (branding) + el patrón de `team-pdf`/`match-pdf`. Estructura = hoja del anexo: **cabecera** (equipo · fecha · sesión · obj. físico/táctico/técnico · tiempo total · meso/micro) + **bloques** en orden, cada uno con sus **tareas** (nombre del ejercicio + duración/series del día + objetivo/incidir-en del ejercicio). **Sin diagramas** en v1 (follow-up: rasterizar `<DiagramView>` a PNG e incrustarlo). Staff full; (si se decide) familia su versión.
