## 0. Resumen ejecutivo

F7 entrega al cuerpo técnico una **pantalla de captura en vivo** para registrar todo lo que pasa durante el partido desde un dispositivo en la banda: arrastra símbolos (gol, asistencia, tarjeta, córner, falta, fuera de juego, tiro) sobre los jugadores o sobre el campo, registra sustituciones, eventos del rival, controla un cronómetro avanzado (descanso, prórroga, ajuste manual), ve en directo los **minutos jugados por cada jugador** y, al pitido final, **cierra el partido consolidando las estadísticas por jugador** que alimentarán el perfil deportivo de F9.

Reutiliza la fundación visual de F6 (`<MatchFieldEditor>`) en `mode='live-overlay'` — ver [ADR-0009](../decisions/ADR-0009-f6-f7-match-field-editor-compartido.md), ya **Accepted**. F7 **no rehace** el componente: le superpone su capa propia (cronómetro, paleta de eventos, timeline) vía `children` + los callbacks `onPlayerClick`/`onFieldClick`.

Esta spec recoge las **decisiones de diseño cerradas** (§2–§9), el modelo de datos completo (§3), el plan de lotes (§11) y los ADR candidatos (§12). Las decisiones marcadas como cerradas en el encargo van como **definitivas**, no "a confirmar".

**Pre-requisito**: **F6 cerrada** (6.1–6.10 ☑, Lote B' incluido — PR #34). ✔ Verificado en [progress.md](../journey/progress.md). El componente `<MatchFieldEditor>` y el catálogo de formaciones de `packages/core` existen.

## 2. Objetivos

- **Principal y verificable**: el cuerpo técnico del equipo puede registrar **todos** los eventos del partido (propio y rival) con un gesto de arrastrar y soltar, desde un Chromebook/tablet táctil en horizontal, y al cerrar el partido se vuelcan las stats por jugador. La línea de tiempo es editable y el cronómetro sobrevive a una recarga de página.
- Ver en directo los **minutos jugados por jugador** con aviso de "ha jugado poco" para repartir minutos.
- No perder lo registrado ante un corte breve de WiFi (resiliencia offline mínima).

## 3. Decisión 1 — Modelo de datos

Cinco tablas, todas ancladas al `event` (el partido), no a `lineups` (que es un plan mutable). Convenciones heredadas: `club_id` denormalizado para multi-tenant, validación por **trigger** (no CHECK con función) cuando depende de `events`/`team_members`, helper RLS `security definer`, `set_updated_at`, y la lección **MVCC** de NIDO (los helpers leen `events`/`team_staff`/`memberships`, ninguna mutada por los INSERT a las tablas de F7 → el `RETURNING *` pasa la policy SELECT sin tropezar; ver [feedback_nido_rls_mvcc_gotcha]).

### 3.1 `match_state` — sesión de captura (1:1 con el partido)

```sql
create table public.match_state (
  event_id        uuid primary key references public.events(id) on delete cascade,
  club_id         uuid not null references public.clubs(id) on delete cascade,  -- denormalizado

  status          text not null default 'not_started' check (status in (
                    'not_started','live','closed')),

  goals_for       smallint check (goals_for is null or goals_for >= 0),   -- marcador final (al cerrar)
  goals_against   smallint check (goals_against is null or goals_against >= 0),

  post_match_notes text check (post_match_notes is null or char_length(post_match_notes) <= 4000),  -- 7.11

  -- Mono-operador (decisión §5.5): lock ADVISORY, no control de concurrencia duro.
  operator_profile_id uuid references public.profiles(id),
  lock_heartbeat_at   timestamptz,

  reopened_count  smallint not null default 0,   -- nº de veces reabierto (decisión §5.3)
  started_at      timestamptz,                   -- pitido inicial real
  closed_at       timestamptz,
  closed_by       uuid references public.profiles(id),

  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);
```

`status`: `not_started` (sesión creada, sin pitido) → `live` (partido en curso) → `closed` (cerrado, stats materializadas). **Reabrir** (§5.3) vuelve a `live`, incrementa `reopened_count` y limpia `closed_at`/`closed_by`.

### 3.2 `match_periods` — reloj persistente (decisión §6)

El cronómetro **vive en BD**, no solo en memoria del cliente, para sobrevivir a recargas (decisión §6). Una fila por periodo jugado:

```sql
create table public.match_periods (
  id              uuid primary key default gen_random_uuid(),
  event_id        uuid not null references public.events(id) on delete cascade,
  period          text not null check (period in (
                    'first_half','second_half','extra_first','extra_second','penalties')),
  ordinal         smallint not null,   -- orden de juego (1,2,3…) para el reloj absoluto

  -- Modelo de reloj recuperable:
  base_offset_seconds  integer not null default 0,   -- a qué segundo absoluto arranca el marcador de este periodo
  accumulated_seconds  integer not null default 0,    -- juego acumulado en este periodo mientras está en PAUSA
  running              boolean not null default false, -- ¿el cronómetro corre ahora?
  last_started_at      timestamptz,                    -- wall-clock del último arranque (null si en pausa)
  ended                boolean not null default false,

  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),

  unique (event_id, period),
  unique (event_id, ordinal)
);
```

**Reloj absoluto recuperable** — `clock_seconds` de un instante = `base_offset_seconds + accumulated_seconds + (running ? now() - last_started_at : 0)`. Tras una recarga, el cliente reconstruye el cronómetro leyendo `match_periods` (no depende de estado en memoria). El **descanso** = todos los periodos en pausa/cerrados y ninguno `running`. La prórroga = periodos `extra_*` con su `ordinal`. El **ajuste manual** edita `accumulated_seconds`/`base_offset_seconds`. La duración sugerida de cada parte sale de `categories.half_duration_minutes` (no se impone: el operador puede prolongar).

### 3.3 `match_starters` — once inicial congelado (decisión §7)

Snapshot del 11/7/8 que **empieza** el partido, **congelado al pitido inicial** — no se lee la alineación de F6 en vivo (puede editarse después). Es la base del cálculo de minutos jugados.

```sql
create table public.match_starters (
  event_id      uuid not null references public.events(id) on delete cascade,
  player_id     uuid not null references public.players(id) on delete cascade,
  position_code text,        -- slot en el que arrancó (informativo; del preset de F6 si vino de lineup)
  created_at    timestamptz not null default now(),
  primary key (event_id, player_id)
);
```

Al pulsar "Iniciar partido" (status → `live`), se congela: si existe alineación oficial (`lineups.is_official` con sus `lineup_positions.location='field'`) se precarga; si no (p. ej. amistoso sin alineación), el operador selecciona el once manualmente desde el roster histórico (`team_members` a la fecha). El resto de convocados quedan como suplentes disponibles para 7.5.

### 3.4 `match_events` — el log del partido (corazón de F7)

```sql
create table public.match_events (
  id            uuid primary key default gen_random_uuid(),   -- el CLIENTE lo genera (idempotencia de reintento §10)
  event_id      uuid not null references public.events(id) on delete cascade,   -- el partido
  club_id       uuid not null references public.clubs(id) on delete cascade,    -- denormalizado multi-tenant

  side          text not null check (side in ('own','rival')),
  type          text not null check (type in (
                  'goal','assist','yellow_card','red_card',
                  'substitution','corner','foul','offside','shot')),

  -- Actor: own → player_id (FK). rival → rival_dorsal (sin roster del rival, decisión §8).
  player_id     uuid references public.players(id) on delete set null,
  rival_dorsal  smallint check (rival_dorsal is null or rival_dorsal between 1 and 99),

  -- Sustitución (y enlace asistencia↔gol): segundo jugador implicado.
  --   substitution: player_id = SALE, related_player_id = ENTRA.
  related_player_id uuid references public.players(id) on delete set null,

  -- Tiempo: periodo + minuto de marcador (display) + SEGUNDOS ABSOLUTOS de juego (cálculo fiable §6).
  period         text not null default 'first_half' check (period in (
                   'first_half','second_half','extra_first','extra_second','penalties')),
  display_minute smallint check (display_minute is null or display_minute between 0 and 130),
  clock_seconds  integer not null check (clock_seconds >= 0),

  -- Ubicación en el campo (solo eventos sobre césped, 7.4): 0–100, equipo atacando hacia arriba (coherente con F6).
  x_pct         numeric(5,2) check (x_pct is null or (x_pct between 0 and 100)),
  y_pct         numeric(5,2) check (y_pct is null or (y_pct between 0 and 100)),

  metadata      jsonb not null default '{}'::jsonb,   -- extras: shot_on_target, card_reason, goal_event_id (asistencia)…

  created_by    uuid not null references public.profiles(id),
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),

  -- Actor coherente con el bando.
  constraint match_events_actor_by_side check (
    (side = 'own'   and rival_dorsal is null)
    or (side = 'rival' and player_id is null and related_player_id is null)),

  -- related_player_id solo en sustituciones (jugador que entra).
  constraint match_events_related_only_sub check (
    related_player_id is null or type = 'substitution'),

  -- Coordenadas solo en eventos de campo.
  constraint match_events_coords_field_only check (
    (x_pct is null and y_pct is null) or type in ('corner','foul','offside','shot'))
);

create index match_events_event_clock_idx on public.match_events (event_id, clock_seconds);  -- timeline + cálculo minutos
create index match_events_player_idx on public.match_events (player_id) where player_id is not null;  -- F9
```

`id` lo **genera el cliente** (UUID v4 en local): al reintentar un INSERT tras un corte de red, el mismo `id` deduplica (idempotencia, §10).

#### 3.4 bis — Regla de expulsión (añadida en 7.3, 2026-06-02)

Reglas de tarjetas/expulsión sobre los eventos de jugador (motor puro `resolveCardOutcome` en `packages/core/src/match/event.ts`, validado además en la server action; no añade columnas — el modelo de 7.1 ya tiene `yellow_card`/`red_card`):

- **2ª amarilla** al mismo jugador ⇒ **expulsión automática**: se registra la amarilla **y** una `red_card` (con `metadata.reason = 'second_yellow'`).
- **2ª roja** al mismo jugador ⇒ **bloqueada**.
- Un jugador **con roja queda expulsado**: SALE del campo (deja de ser seleccionable) y **no puede recibir más eventos** (ni gol/asistencia/amarilla/roja).
- La `red_card` en `match_events` es la fuente de verdad: **7.8** dejará de sumarle minutos (§6: `red_card` es una "salida") y **7.5** impedirá que vuelva por sustitución. 7.3 solo deja la expulsión registrada y al jugador fuera del campo (mostrarlo en un banquillo "como expulsado" llega con 7.5).

### 3.5 `match_player_stats` — consolidación al cierre (insumo de F9, decisión §9)

Materializada **solo al cerrar** el partido. Una fila por jugador propio que participó. **F7 no construye el perfil agregado** (suma de temporada, ratios, evolución) — eso es F9; F7 entrega este insumo por-partido.

```sql
create table public.match_player_stats (
  event_id        uuid not null references public.events(id) on delete cascade,
  player_id       uuid not null references public.players(id) on delete cascade,
  club_id         uuid not null references public.clubs(id) on delete cascade,   -- denormalizado
  team_id         uuid not null references public.teams(id) on delete cascade,

  started         boolean not null default false,
  minutes_played  smallint not null default 0,   -- derivado del reloj + subs + rojas + fin de periodo
  goals           smallint not null default 0,
  assists         smallint not null default 0,
  yellow_cards    smallint not null default 0,
  red_cards       smallint not null default 0,
  shots           smallint not null default 0,
  fouls           smallint not null default 0,

  computed_at     timestamptz not null default now(),
  primary key (event_id, player_id)
);

create index match_player_stats_player_idx on public.match_player_stats (player_id);  -- F9 agrega por aquí
```

Al cerrar: se calcula desde `match_events` + `match_starters` + `match_periods` y se inserta. Al **reabrir → editar → re-cerrar**: se **borra y recalcula** la cara del partido (`delete where event_id` + reinsert) → consistente, sin filas obsoletas (decisión §5.3).

## 4. Decisión 2 — Permisos (RLS)

**Decisión cerrada (§5.1): la captura en vivo la realiza el cuerpo técnico del equipo de ese partido — `team_staff` activo (entrenador principal Y ayudante) — NO solo quien creó la alineación.** Resuelto por **autoridad de `team_staff`** (patrón F4), no por `memberships.role`.

**Capability propia `can_record_match` vs reusar autoridad de staff — decisión: NO se crea capability nueva.** Gate = **cualquier `team_staff` activo del equipo del partido** (principal o ayudante, vía `user_is_staff_of_team`) + admin/coord del club. Razonamiento:

- A diferencia de F4 (que sí creó `can_manage_callups` porque "gestionar convocatoria" no encajaba en ninguna capability), aquí el caso real es **"el que está en la banda registra"**, y ese puede ser cualquiera del cuerpo técnico del equipo. Exigir una capability a los ayudantes **bloquearía el caso común** (el ayudante que lleva la tablet) y obligaría a un seed/backfill nuevo.
- `user_is_staff_of_team(team_id)` ya devuelve TRUE para cualquier `team_staff` activo (principal o ayudante) — encaja exacto, sin lógica nueva.
- Se documenta `can_record_match` como **palanca futura** (estilo F14) si algún club quiere restringir qué ayudantes registran. Hoy es overhead innecesario.

Helper nuevo (espejo de `user_can_manage_lineup`, con la lección "autoridad por equipo" ya incorporada vía `user_is_staff_of_team`):

```sql
create or replace function public.user_can_record_match(p_event_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select
    public.user_role_in_club(e.club_id) in ('admin_club','coordinador')
    or (e.team_id is not null and public.user_is_staff_of_team(e.team_id))
    from public.events e where e.id = p_event_id;
$$;
```

RLS de las 5 tablas:
- **`match_state`, `match_periods`, `match_starters`, `match_events`, `match_player_stats`**: SELECT/INSERT/UPDATE/DELETE ligadas a `user_can_record_match(event_id)`. Sin gotcha MVCC (el helper lee `events`/`team_staff`, no las tablas de F7).
- **Lectura ampliada futura**: que jugador/familia vean *sus* stats del partido se evalúa en **F9** (perfil del jugador), no en F7. F7 es staff-only en lectura.

## 5. Decisión 3 — Reglas de operación (decisiones cerradas)

### 5.1 Quién registra
Cuerpo técnico del equipo (`team_staff` activo: principal + ayudante) + admin/coord. Ver §4.

### 5.2 Qué partidos
**`type='match'` Y `type='friendly'`.** Los amistosos también se registran. Ajuste de gating: relajar `lineups_validate` a `type in ('match','friendly')` (§1, migración F7.1) para que el once inicial pueda venir de una alineación de amistoso; el trigger de `match_events` valida `type in ('match','friendly')`.

### 5.3 Reabrir partido
- **Cerrar** materializa `match_player_stats` y pone `status='closed'`.
- **Reabrir** es un paso **explícito** (botón con confirmación, no accidental): `status` vuelve a `live`, `reopened_count += 1`.
- Editar (vía timeline 7.9) y **volver a cerrar** → `match_player_stats` se **recalcula de forma consistente** (delete+reinsert de la cara del partido). La corrección en vivo ya la cubre la línea de tiempo editable; reabrir es para correcciones **tras** el cierre.

### 5.4 Resiliencia offline (MÍNIMA)
Estado local optimista + cola de reintento (ver §10). Cubre **cortes breves** de WiFi sin perder lo registrado. **Offline completo** (partido entero sin red) queda **FUERA de F7** → futuro (§10 / Fuera de alcance).

### 5.5 Mono-operador
**Un solo dispositivo/persona por partido.** Sin concurrencia multi-usuario, sin merge CRDT. `match_state.operator_profile_id` + `lock_heartbeat_at` son un **lock advisory**: si otro staff abre la captura del mismo partido, la UI **avisa** ("ya hay una sesión activa en otro dispositivo") y ofrece tomar el control, pero la BD no impone exclusión dura. Multi-operador colaborativo → futuro.

## 6. Decisión 4 — Reloj y minutos fiables

El **minuto de marcador no basta** para sumar minutos jugados con descanso, prórroga y cambios. Por eso cada `match_event` guarda **`clock_seconds` absoluto** (segundos de juego transcurridos desde el pitido inicial, monótono no decreciente, el descanso no cuenta), además del `display_minute`/`period` que se muestran.

- **Origen del reloj**: `match_periods` (§3.2). El cronómetro es recuperable tras recarga (fórmula en §3.2). Eso resuelve a la vez la **persistencia/recuperación** y el **insumo del cálculo de minutos**.
- **Cálculo de minutos jugados** (función pura en `packages/core/src/match/`, testeable sin DOM):
  `minutos(jugador) = Σ (instante_salida − instante_entrada)` sobre los intervalos en campo, en `clock_seconds`, donde:
  - **entrada** ∈ {pitido inicial si era titular (`match_starters`), `substitution` en el que `related_player_id` = jugador};
  - **salida** ∈ {`substitution` en el que `player_id` = jugador, `red_card` del jugador, fin del último periodo jugado}.
- Robusto ante descanso (el reloj no avanza entre periodos), prórroga (periodos `extra_*` con su `ordinal`) y expulsión (deja de acumular). No depende del minuto de marcador.

## 7. Decisión 5 — `<MatchFieldEditor mode='live-overlay'>` (cableado)

F7 monta el componente de F6 en `mode='live-overlay'`. El modo está **tipado** en la API ([6.0-alineaciones.md §4](6.0-alineaciones.md)) pero F6 solo necesitó `edit`/`readonly` → su cableado real es el **"<2 h de refactor" previsto en ADR-0009** y va como **subtarea explícita de 7.2** (no se asume hecho). Verificar/implementar:

1. **`children` como overlays absolutos** posicionados sobre el campo (cronómetro, paleta de símbolos, panel rival).
2. **`onFieldClick(xPct, yPct)`** devuelve coordenadas 0–100 con el mismo sistema que F6 (equipo atacando hacia arriba) → alimenta `x_pct`/`y_pct` de 7.4.
3. **`onPlayerClick(playerId)`** para asignar evento al jugador (7.3).
4. **Drag&drop desde fuera**: arrastrar un chip de la paleta y soltarlo sobre un jugador o sobre el césped (cursor de coordenadas). Es el gesto central de captura; si la implementación de F6 solo soporta drag interno (campo↔banquillo), aquí se añade el "drop externo".
5. **Sin lógica de eventos de partido dentro del componente** — sigue conociendo solo jugadores/posiciones; F7 aporta la semántica.

Si el refactor excede lo previsto, se anota en notas de implementación y se evalúa ADR.

### 8 bis — Panel de próximo partido en Inicio (7.12)

Subfase transversal (no es captura en vivo; es la **puerta de entrada** desde Inicio al flujo del partido). Lee estado de F4/F6/F7.1, sin migración.

- **Cuerpo técnico** (entrenador principal/ayudante, de SU equipo, vía `team_staff` activo): tarjeta del **próximo partido** (el más cercano `starts_at >= now`, `type` match/friendly) con estado y un único botón al **paso que toca**, en este orden de prioridad:
  1. `match_state.status = 'closed'` → **"Postpartido pendiente"** → placeholder deshabilitado (F8 no existe aún).
  2. existe `lineups.is_official = true` → **"Preparado para el partido"** → **"En directo"** (`/convocatorias/[id]/directo`).
  3. convocatoria publicada y **todos los convocados han respondido** (X≥Y) → **"Falta la alineación oficial"** → editor (`/convocatorias/[id]/alineacion`).
  4. convocatoria publicada, faltan respuestas → **"Esperando confirmación (X/Y)"** → convocatoria (`/convocatorias/[id]`).
  5. sin `match_callup_meta.published_at` → **"Preparar convocatoria"** → convocatoria.
- **X/Y**: Y = convocados = roster a la fecha − descartados (`callup_decisions.discarded`); X = convocados que **han respondido** (yes/maybe/no) en `callup_responses`. "Confirmación" = ha respondido (no solo 'yes'), para que un 'no'/'duda' no bloquee el avance.
- **Jugador / familia**: si tiene un jugador propio convocado (no descartado) en una convocatoria **publicada** próxima y **sin responder** → aviso "Tienes una convocatoria pendiente de confirmar" + botón **"Confirmar disponibilidad"** (`/convocatorias/[id]`). Nada más.
- **Admin/coord**: NO se muestra (el resumen global llega con F9).
- Muestra **solo el próximo** partido. No auto-marca oficial ninguna alineación ni hace fallback a "la última".

## 9. Decisión 6 — Consolidación al cierre (7.10)

- **Estado actual**: no existe tabla de stats agregadas; el perfil deportivo agregado es **F9**. F7 **solo** produce el insumo por-partido `match_player_stats` (§3.5).
- **Materializar al cerrar, no derivar en lectura.** Motivos: (1) **rendimiento de F9** — los perfiles agregan muchos partidos; sumar `match_events` crudos en cada visita escala mal; (2) **inmutabilidad/cierre** — congela el resultado, `match_events` queda como log de auditoría; (3) **des-arriesga F9** — le da un contrato de entrada estable. → **ADR-0015** (§12).
- Solo `side='own'` se vuelca. Eventos del rival y "rivales destacados" (7.11) no afectan a ningún `match_player_stats`.
- `minutes_played` se calcula con el motor de §6. Recalculable e idempotente al reabrir (delete+reinsert).

## 10. Decisión 7 — Resiliencia offline mínima (7.2, transversal a Lote A)

- **Estado local optimista**: al registrar un evento, la UI lo pinta de inmediato y lo encola.
- **Cola de reintento**: cada evento lleva su `id` (UUID generado en cliente). Se persiste la cola en almacenamiento local (IndexedDB/localStorage) y se reintenta el INSERT con backoff hasta confirmar. Como el `id` es estable, el reintento es **idempotente** (un `insert ... on conflict (id) do nothing` evita duplicados).
- **Indicador de sincronización** en la UI: "N eventos pendientes de guardar".
- **Fuera de F7**: funcionar un **partido entero sin red** (arranque offline, sin sesión cargada). Eso exigiría sincronización bidireccional y resolución de conflictos → futuro.
