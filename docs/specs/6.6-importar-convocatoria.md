# Spec 6.6 — Importar plantilla desde convocatoria F4

> Subfase del Plan Maestro: **6.6 — Importar plantilla de convocatoria F4**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-29 (bosquejo)

---

## 1. Contexto

F6 prepara la alineación de un partido. El estado de cada jugador (disponible / duda / no disponible / descarte técnico) ya existe en F4 a través de las tablas `match_callup_meta` (publicación), `callup_responses` (yes/maybe/no del jugador o familia) y `callup_decisions` (called_up/discarded del cuerpo técnico). En lugar de pedir al entrenador que vuelva a marcar lo que ya ha sido decidido en F4, esta subfase importa esa información como punto de partida del editor de F6.

**Dependencias**: F4 cerrada (`match_callup_meta`, `callup_responses`, `callup_decisions` y sus capabilities/RLS estables).

## 2. Objetivos

- Al abrir el editor de F6 para un partido cuya convocatoria está publicada, los jugadores aparecen pre-clasificados en:
  - **disponible** — respuesta `yes` + decisión `called_up` (o sin decisión técnica).
  - **duda** — respuesta `maybe` + decisión `called_up` (o sin decisión técnica).
  - **no disponible** — respuesta `no`.
  - **descarte técnico** — decisión `discarded` (con prioridad sobre la respuesta del jugador).
- Para jugadores del roster sin respuesta y sin decisión, estado `sin_dato`.
- El estado importado se persiste como snapshot en F6 (no se vuelve a leer cada render — protege ante cambios posteriores en F4).
- El entrenador puede **reimportar** desde un botón explícito si quiere refrescar el snapshot.

## 3. Fuera de alcance

- **NO** se modifica F4 desde F6. La lectura es unidireccional `F4 → F6`. Cambiar la respuesta del jugador o la decisión técnica sigue ocurriendo en `/convocatorias/[eventId]`.
- **NO** se sincroniza automáticamente: si la convocatoria cambia después de importar, el snapshot de F6 se queda con lo importado. El reimport explícito es la única vía de actualización.
- **NO** se importan respuestas de partidos sin convocatoria publicada (estado `draft` en `match_callup_meta` no expone respuestas a este flujo).

## 4. Modelo de datos afectado

Tabla nueva o columna nueva sobre `lineup_positions` (a decidir en implementación):

```sql
-- Propuesta A (columna):
alter table public.lineup_positions
  add column callup_status text
    check (callup_status in (
      'disponible',
      'duda',
      'no_disponible',
      'descarte_tecnico',
      'sin_dato'
    )),
  add column callup_imported_at timestamptz;
```

```sql
-- Propuesta B (tabla aparte, si lineup_positions se carga sólo con titulares):
create table public.lineup_callup_snapshot (
  lineup_id   uuid not null references public.lineups(id) on delete cascade,
  player_id   uuid not null references public.players(id),
  callup_status text not null,
  imported_at timestamptz not null default now(),
  primary key (lineup_id, player_id)
);
```

Decisión final en la implementación (probable: propuesta B, porque F6.7 introduce banquillo + fuera convocatoria y el snapshot debe cubrir todo el roster, no solo titulares).

## 5. Flujo

1. Entrenador abre `/equipos/[teamId]/partidos/[matchId]/lineup`.
2. Si no hay snapshot importado → CTA "Importar convocatoria".
3. Server action lee `callup_responses` + `callup_decisions` del evento + roster histórico de `team_members` a la fecha del partido (mismo patrón snapshot que F4.3) → calcula `callup_status` por jugador → escribe el snapshot.
4. Editor renderiza con cada jugador en su estado importado.
5. Botón "Reimportar" disponible mientras el partido sea futuro.

## 6. Permisos

- Lectura `callup_responses`/`callup_decisions`: limitada por RLS de F4 (la query la hace el server action con la sesión del coach, así que la RLS ya filtra).
- Escritura snapshot: `can_manage_lineup` (capability de F6, ver F6.1) o principal del team_staff (autoridad por equipo, ver fix F4 Lote B 2026-05-29).

## 7. Tests

- Vitest puro del mapeo `(response, decision) → callup_status` con todos los pares posibles.
- pgTAP: snapshot escrito por principal del team_staff (no por el rol del club).
- Smoke manual: convocatoria con 3 yes / 1 maybe / 1 no / 1 discarded → editor muestra 3+1 disponibles+duda, 1 no, 1 descarte. Reimport tras cambio de respuesta refleja el cambio.
