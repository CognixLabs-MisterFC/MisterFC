## 1. Contexto

Cuando un jugador (o familia) entra a la app hoy, el rol jugador NO tiene un "home del equipo". El sidebar le ofrece Inicio, Calendario, Asistencia, Convocatorias, Mensajes, Mi perfil. La home rol-aware (de F5 Lote A) muestra cards transversales (mensajes, anuncios, próximos eventos), pero no hay una pantalla específica con "mi equipo, mis compañeros, mis próximos eventos, mis anuncios" todo junto.

F5.8 cubre exactamente ese vacío: una sola pantalla `/es/mi-equipo` que es el espejo de `/mis-equipos` (que es para coaches multi-equipo). Para el jugador es "mi equipo" en singular en la mayoría de casos (un jugador suele estar en 1 team).

## 2. Objetivos

- Jugador entra a `/mi-equipo` y ve, sin scrollar: equipo, compañeros, próximos eventos, anuncios recientes.
- Acceso 1-click a `/convocatorias` (su lista personal de partidos por confirmar).
- Si el jugador está en varios teams (pretemporada, varias categorías), selector de equipo arriba.

## 3. Fuera de alcance

- **Stats personales / evolución / reportes** → eso es F9 (perfil del jugador con gráficas).
- **Posición en el campo / pizarra** → F6/F7.
- **Chat de grupo del equipo** → no planificado en MisterFC; canales son 1:1 (mensajes) + broadcast (anuncios).
- **Acciones del jugador sobre compañeros** (añadir, marcar, etc.) → solo lectura.

## 4. Decisiones

### D1. Solo rol jugador

`/es/mi-equipo` redirige a `/` para cualquier rol distinto de `jugador`. Otros roles ya tienen sus vistas (`/mis-equipos` para coaches, `/jugadores` para admin/coord). El sidebar item solo aparece para jugador.

### D2. Reuso de modelo, cero migración

Toda la data viene de tablas existentes: `team_members` + `players` + `events` + `announcements` + `categories` (para `half_duration_minutes`). RLS existente filtra: jugador ve sus teams (via `team_members` + `player_accounts`).

### D3. Multi-team con dropdown

Caso poco común pero real: un jugador puede estar en 2 teams (pretemporada A + benjamín B). Cargamos el(los) team_id(s) del jugador y, si >1, mostramos `<Select>` arriba. El parámetro queda en URL (`?team=...`) para que el back/forward funcione.

### D4. Anuncios: team + club-wide juntos

La query pide `announcements` del club con `team_id IN (player_teams) OR team_id IS NULL`. RLS ya filtra, pero el query explícito lo hace claro y limita filas. Orden: pinned desc → created_at desc, limit 5.

### D5. Próximos eventos

Eventos del team con `starts_at >= now()` y `<= now() + 30d`, orden ASC, limit 10. Reusa la query patrón de `/calendario`.

## 5. Modelo

Sin cambios de schema. Helpers nuevos:

```ts
// packages/core/src/team-view/helpers.ts
export function listTeammates(players, currentPlayerId): PlayerCard[] // excluye al propio jugador
export function listUpcomingTeamEvents(events, horizonDays = 30): EventCard[]
export function listVisibleAnnouncements(announcements, teamIds, limit = 5): AnnouncementCard[]
```

Funciones puras (sin BD, sin Date.now()). Las llama el server component tras el SELECT.

## 6. UI

`/es/mi-equipo`:
- Header: nombre del equipo + categoría + temporada + duración por tiempo (informativo, p.ej. "Alevín · 30 min/tiempo").
- Si multi-team: dropdown.
- 4 cards:
  1. **Compañeros** — grid de avatars con nombre + dorsal (10-15 max visibles, "ver todos" si más).
  2. **Próximos eventos** — lista cronológica con tipo (entrenamiento/partido), hora, lugar.
  3. **Anuncios recientes** — lista (igual al estilo de home card).
  4. **Mis convocatorias** — link a `/convocatorias`.

Sidebar: nuevo item `mi_equipo` solo para `role=jugador`. Icono: `Shield` (igual que `/mis-equipos` del coach pero en singular).

## 7. Tests

### Vitest (helpers puros)
- `listTeammates`: excluye el current player, orden alfabético por full_name.
- `listUpcomingTeamEvents`: filtra por horizon, ordena ASC, limit configurable.
- `listVisibleAnnouncements`: pinned primero, team-bound + club-wide, dedupe por id.

### Sin pgTAP nuevo
- RLS de `team_members` + `players` + `events` + `announcements` ya cubierta por tests existentes. No introducimos policies nuevas.
