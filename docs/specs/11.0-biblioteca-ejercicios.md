## 0. Resumen ejecutivo

F11 es el **sistema de gestión de ejercicios de entrenamiento del club**: explorar (catálogo + filtros), ver ficha (campos + diagrama del campo), crear los propios con un **editor visual (PitchEditor)**, y un **ciclo de aprobación** (entrenador propone → admin publica) que convierte un ejercicio en parte de **la metodología del club** (visible/usable por todos los entrenadores).

Es la **base de contenido** que consumen **F12** (planificador de sesiones, arrastra ejercicios) y **F13** (jugadas animadas), y cuya **capa de dibujo** reutiliza **F11B** (pizarra en vivo). Dos piezas son contratos de larga vida y se diseñan aquí con cuidado:

1. **El esquema del diagrama** (`@misterfc/core`, Zod puro, frame-extensible) — lo consumen F11B/F12/F13.
2. **El ciclo de metodología** (borrador → propuesto → publicado/rechazado + archivado) — lo reutilizará F12 para las plantillas de sesión.

La mayor parte de F11 es **aditiva** (tabla `exercises` nueva, sin datos previos); el único cambio sobre estructura existente es **ampliar el set de capabilities** (EXPAND del CHECK + `CAPABILITY_NAMES` + backfill) para añadir `can_create_exercises`, con el patrón ya usado en F3/F4.

> **Nota de fidelidad ⚠️** — Los campos del ejercicio y el enum de elementos del diagrama se han derivado de la **hoja de sesión real** adjuntada por el cliente (Infantil B, 11/06/2026). El mapeo verbatim anexo↔modelo está en el **§13 (Anexo)**. La estructura de **sesión** (cabecera meso/microciclo + bloques) de esa misma hoja es material para la **spec de F12**, no para F11.

## 2. Objetivos y criterio de cierre

**Objetivo**: el cuerpo técnico explora una biblioteca de ejercicios del club con filtros (categoría de edad, **objetivo táctico y técnico** —dos ejes—, intensidad, espacio, duración), ve la ficha completa (campos + diagrama), crea los suyos con el editor visual, y los **propone**; el **Admin** los aprueba para que pasen a ser **metodología del club** (visibles/usables por todos los entrenadores).

**Criterio de cierre (F11 v1)**:
- Un entrenador con `can_create_exercises` crea un ejercicio (con o sin diagrama) → queda **borrador/propuesto**.
- El Admin ve los **propuestos**, **aprueba** (→ publicado) o **rechaza** (→ rechazado, con motivo + aviso al autor).
- Los **publicados** se listan/filtran y se ven en ficha por **todo el staff del club**.
- Import/export de un ejercicio (fichero JSON, incluye diagrama).
- `typecheck · lint · test · build` + pgTAP en verde en cada subfase.

## 4. Modelo de datos

### 4.1 Tabla `exercises` (nueva — calca `coach_formations` + máquina de estados)

> DDL **ilustrativo** (la migración real se escribe en 11.1). Atributos de dominio en español (vocabulario de la metodología, fieles a la hoja real); ciclo de vida/auditoría en inglés (convención del repo: `owner_profile_id`, `created_at`, `approved_by`…).

```sql
create table public.exercises (
  id                 uuid primary key default gen_random_uuid(),
  owner_profile_id   uuid not null references public.profiles(id) on delete cascade,
  club_id            uuid not null references public.clubs(id)    on delete cascade,
  name               text not null check (char_length(name) between 1 and 120),

  -- Campos de la TAREA (anexo §13). Columnas en INGLÉS (convención del repo);
  -- los VALORES de las taxonomías van en español (vocabulario de la metodología).
  description         text,        -- "DESCRIPCIÓN DE REGLAS-NORMAS"
  objective           text,        -- "OBJETIVO DE LA TAREA / DE LA CHARLA"
  coaching_points     text,        -- "INCIDIR EN" (texto libre)
  variants            text,        -- "VARIANTE 1…" (texto libre)
  players             text,        -- "18 Jugadores (3 equipos de 6)" (texto libre)

  -- Taxonomías (D4 — modelo B: dos ejes estructurados + físico libre):
  categories           text[] not null default '{}',  -- subconjunto de CATEGORY_KINDS (multi)
  tactical_objectives  text[] not null default '{}',  -- vocabulario controlado (multi, §4.7)
  technical_objectives text[] not null default '{}',  -- vocabulario controlado (multi, §4.7)
  physical_focus       text,        -- texto libre, opcional (no estructurado en v1)
  intensity            text check (intensity is null or intensity in ('baja','media','alta')),
  space_type           text check (space_type is null or space_type in
                         ('campo_completo','medio_campo','cuarto_campo','reducido')),
  space_dimensions     text,        -- "25x10" (libre, opcional)
  base_duration        smallint check (base_duration is null or base_duration >= 0), -- minutos

  -- Diagrama (opcional, D5). Validado por trigger como coach_formations.positions:
  diagram            jsonb,

  -- Máquina de estados (D3):
  status             text not null default 'draft' check (status in
                       ('draft','proposed','published','rejected')),
  approved_by        uuid references public.profiles(id),
  approved_at        timestamptz,
  rejection_reason   text,
  archived_at        timestamptz,  -- soft-delete de publicados (referencias F12)

  created_at         timestamptz not null default now(),
  updated_at         timestamptz not null default now()
);

create index exercises_club_status_idx on public.exercises (club_id, status)
  where archived_at is null;
create index exercises_owner_idx on public.exercises (owner_profile_id);
```

**Validaciones por trigger** (`exercises_validate`, patrón `coach_formations_validate`):
- `owner_profile_id := auth.uid()` en INSERT; `owner_profile_id`/`club_id` inmutables en UPDATE; `updated_at := now()`.
- `categories[]` ⊆ `CATEGORY_KINDS`; `tactical_objectives[]` y `technical_objectives[]` ⊆ sus vocabularios controlados (§4.7); `physical_focus` es texto libre.
- `diagram` (si no null): valida la forma del esquema (§4.2) — array de elementos tipados con coords en `[0,100]`.
- **Reglas de transición de estado** (§4.4) que dependen del rol → en el trigger (no en CHECK, porque consultan rol).

### 4.2 Esquema del diagrama (subfase 11.0 — **contrato crítico**)

Vive en **`@misterfc/core`** (Zod puro + tests, sin DOM). Una **escena estática**:

```ts
// DiagramScene v1 — F11 guarda UNA escena. F13 la envolverá en frames[] (ver abajo).
type Pct = number;               // 0–100
type Point = { x_pct: Pct; y_pct: Pct };

type DiagramElement =
  | { type: 'jugador'; id: string; x_pct: Pct; y_pct: Pct;
      rol: 'atacante' | 'defensor' | 'comodin' | 'portero'; label?: string }
  | { type: 'balon';          id: string; x_pct: Pct; y_pct: Pct }
  | { type: 'cono';           id: string; x_pct: Pct; y_pct: Pct }
  | { type: 'aro';            id: string; x_pct: Pct; y_pct: Pct }
  | { type: 'porteria';       id: string; x_pct: Pct; y_pct: Pct; rotation?: number }
  | { type: 'miniporteria';   id: string; x_pct: Pct; y_pct: Pct; rotation?: number }
  | { type: 'gol_conduccion'; id: string; x_pct: Pct; y_pct: Pct }
  | { type: 'zona'; id: string; x_pct: Pct; y_pct: Pct; w_pct: Pct; h_pct: Pct;
      style: 'solid' | 'dashed' }                          // rect en %, discontinua = carril
  | { type: 'flecha'; id: string; from: Point; to: Point;
      estilo: 'pase' | 'conduccion' | 'desmarque' }
  | { type: 'linea';  id: string; points: Point[] }        // trazo libre / línea de movimiento
  | { type: 'texto';  id: string; x_pct: Pct; y_pct: Pct; text: string } // etiqueta A/B/C…
  | { type: 'cota';   id: string; from: Point; to: Point; label: string }; // medida ("A 40 MTS")

type DiagramScene = { version: 1; elements: DiagramElement[] };
```

**Frame-extensibilidad para F13 (parte del contrato)**: cada elemento tiene un **`id` estable**. F13 **no rediseña el elemento**: envuelve la escena en `frames[]` (p.ej. `{ version, frames: DiagramScene[] }`) e **interpola posiciones por `id`** entre frames. F11 escribe un único frame implícito. La validación de F11 acepta `DiagramScene`; la de F13 aceptará `{ frames: DiagramScene[] }` reutilizando el validador de elemento.

> Enum de elementos = **leyenda de la pág. 3 del anexo** (verbatim): AROS, ZONA DE JUEGO, JUGADOR ATACANTE/DEFENSOR, COMODÍN, PORTERO, MINIPORTERÍA, PORTERÍA, CARRILES/ZONAS (zona discontinua), GOL EN CONDUCCIÓN, FLECHAS (sólida=pase/conducción, discontinua=desmarque), CONOS, COTA ("A 40 MTS"), etiqueta de texto ("A"). `balon` se añade (aparece en los diagramas aunque no esté rotulado en la leyenda).

### 4.4 Máquina de estados (D3)

```
            crea (coach)                 aprueba (Admin)
  ∅ ───────────────────────▶ draft ─(envía)─▶ proposed ─────────────▶ published
  │                                              │  ▲                      │
  │ crea (Admin) → published directo             │  │ edita y reenvía      │ archiva (Admin)
  │                                  rechaza (Admin, motivo) │             ▼
  └───────────────────────────────────────────▶ rejected ──┘          archived
                                                  (aviso al autor)   (archived_at; referenciable)
```

- **draft**: privado del autor. **proposed**: visible para autor + Admin (cola de aprobación). **published** = metodología del club (todo el staff). **rejected**: autor + Admin; el autor edita y reenvía (→ proposed).
- **Admin crea** → entra directo en `published` (salta `proposed`).
- **Borrado** (§ política): `draft`/`proposed`/`rejected` → **borrado duro**; `published` → **archivado** (`archived_at`, soft-delete) para no romper futuras referencias de F12. El autor borra los suyos **no publicados**; el Admin borra/archiva cualquiera.

### 4.5 RLS (adapta `coach_formations` + estados; pgTAP como el resto)

- **SELECT**:
  - `published` (y `archived` con `archived_at`, para que resuelvan referencias de F12) → **todo el staff del club** (cualquier membership con rol de staff en `club_id`).
  - `draft` → **solo el autor**.
  - `proposed` / `rejected` → **autor + Admin** del club.
- **INSERT**: `owner_profile_id = auth.uid()` **y** (`user_has_capability_in_club(club_id,'can_create_exercises')` **o** rol admin/coord). Un no-Admin **no puede** insertar `status='published'` (lo fuerza el trigger: coach entra en `draft`/`proposed`; Admin puede `published`).
- **UPDATE de contenido**: el **autor** mientras `status ∈ (draft, proposed, rejected)`.
- **UPDATE de transición** a `published`/`rejected` (y set de `approved_by`/`approved_at`/`rejection_reason`): **solo Admin**.
- **Archivar** (`archived_at`) un `published`: **solo Admin**.
- **DELETE**: autor si `status ∈ (draft, proposed, rejected)`; Admin cualquiera no publicado (los publicados se archivan, no se borran).

### 4.6 Notificación de rechazo (reusa F5)

`alter type notification_type add value 'exercise_rejected'`. Al rechazar, insertar `notifications` `channel='in_app'`, `user_id = owner_profile_id`, `payload = { exercise_id, rejection_reason }`, `dedupe_key = 'exercise_rejected:'||exercise_id||':'||updated_at`. *(Opcional, fuera de v1: `exercise_published` para avisar de la aprobación — no requerido por la brief.)*

### 4.7 Vocabularios controlados de objetivos (D4 — modelo B)

Dos ejes estructurados (multi-valor, vocabulario controlado) + foco físico libre. **Las columnas van en inglés; los VALORES en español** (vocabulario de la metodología, sin traducir). Viven como constantes en `@misterfc/core` (junto a `CATEGORY_KINDS`), son la fuente del CHECK/validación del trigger de 11.1 y de los Zod de los formularios.

**`tactical_objectives` (20 valores):**
`posesion, salida_de_balon, progresion, ocupacion_del_espacio, lineas_de_pase, cambio_de_orientacion, superioridad, apoyos_y_desmarques, accion_combinativa, amplitud_y_profundidad, juego_por_bandas, centros, finalizacion, presion_tras_perdida, repliegue, basculacion, coberturas_y_vigilancias, transicion_ofensiva, transicion_defensiva, balon_parado`

## 6. Estados, validaciones y errores · Decisiones (estilo "D")

**Estados vacíos**: club sin ejercicios publicados (mensaje + CTA "crear el primero"); autor sin borradores; Admin sin propuestos en cola. **Errores**: rechazo sin motivo (bloquea); editar un `published` (no permitido salvo Admin archivar/duplicar); transición no autorizada → `forbidden` (RLS + mensaje).

### Decisiones cerradas (confirmadas por la brief / el repo)
- **D1 · Catálogo inicial**: **sin seed de fábrica**; lo crea el club (Admin → publicado). La 11.2 queda sin seed.
- **D2 · Capability**: nueva **`can_create_exercises`** (entrenadores/ayudantes); aprobación gateada por **rol Admin**, no por capability.
- **D3 · Estados/visibilidad**: máquina `draft → proposed → published / rejected` + `archived` (soft-delete de publicados). Admin crea directo `published`.
- **D4 · Taxonomías y objetivos (modelo B)**: `categories[]` = **`CATEGORY_KINDS`** (10, multi). Objetivos en **dos ejes estructurados** controlados — `tactical_objectives[]` (20 valores) + `technical_objectives[]` (8 valores) — más `physical_focus` **texto libre** (sin estructurar en v1). `coaching_points` libre. `intensity` enum `baja/media/alta` (opcional). `space_type` enum `campo_completo/medio_campo/cuarto_campo/reducido` + `space_dimensions` libre. `base_duration` entero (min); series/tiempo del día ("2x8'") es **override de F12**, no va en el ejercicio. Vocabularios completos en §4.7.
- **D5 · PitchEditor/diagrama**: SVG nuevo (no extiende `MatchFieldEditor`), renderer read-only compartido, diagrama **opcional**.
- **D6 · Import/export**: individual, **fichero**, incluye diagrama.
- **D7 · Visibilidad a jugadores**: **fuera de F11** (la lleva F12).

> **Naming (cerrado)**: columnas en **inglés** (consistencia con `last_name`/`positions`): `description, objective, coaching_points, variants, players, categories, tactical_objectives, technical_objectives, physical_focus, intensity, space_type, space_dimensions, base_duration` + ciclo/auditoría (`status, approved_by, approved_at, rejection_reason, archived_at`). Los **valores** de las taxonomías van en **español** (metodología), sin traducir.

## 7. Reuso nombrado para F12 — "ciclo de metodología del club"

El patrón **proponer → aprobar/rechazar → publicar + archivar** (máquina de estados §4.4 + RLS por estado §4.5 + aviso al autor §4.6 + soft-delete de publicados) se nombra aquí **"ciclo de metodología del club"** y se diseña **genérico** para que **F12 lo reutilice** en las **plantillas de sesión/microciclo** (entrenador propone una plantilla, Admin la publica como plantilla oficial del club). F12 **no debe rediseñar** este ciclo: lo aplica a su tabla `session_templates`. Si conviene, extraer los helpers de transición/estado a `@misterfc/core` en 11.7 para que F12 los importe.

## 10. Dependencias y riesgos

**Dependencias reales** (más allá del "F2 cerrada" del roadmap): **F6** (`FieldMarkings`/`MatchFieldEditor` para el PitchEditor — *no listada en el roadmap, señalada en §1.2*), **F2.10** (patrón de listado), **F1.4/F1.7** (capabilities + toolkit RLS), **F5** (notificaciones). Precedente **`coach_formations` (F6.10)** reduce mucho el riesgo de modelo+RLS.

**Riesgos**:
1. **El esquema del diagrama es un contrato** que heredan F11B/F12/F13 → si cambia luego, arrastra migraciones de `plays`/`sessions`. Diseñarlo bien (con `id` estable y frame-extensible) **es el trabajo crítico** (11.0).
2. **Scope creep del PitchEditor** (interacción táctil, trazo libre, deshacer) — la pieza de más horas.
3. **i18n de las etiquetas** de los vocabularios (los valores son español canónico; las **labels mostradas** se traducen es/en/va vía claves i18n, no se cambian los valores almacenados).
4. La **capability nueva** toca el CHECK + backfill (bajo riesgo, patrón conocido).
