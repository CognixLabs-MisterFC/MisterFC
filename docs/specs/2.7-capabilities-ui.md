# Spec 2.7 — UI de configuración de capabilities del ayudante

> Subfase del Plan Maestro: **2.7 — UI de configuración de capabilities del ayudante**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-28

---

## 1. Contexto

El modelo está montado desde F1.4: la tabla `public.capabilities` guarda 8 toggles por `membership_id` de entrenador_ayudante. Un trigger `ensure_assistant_capabilities` siembra las 8 filas (granted=false) al crear o cambiar a rol ayudante. RLS (F1.7) permite UPDATE solo a admin/coord/principal del club al que pertenece la membership. Falta la UI para que el entrenador principal de un equipo active/desactive cada capability de cada ayudante.

## 2. Objetivos

- Desde el detalle de un equipo (`/equipos/[teamId]`, introducido en lote A/C), un bloque "Cuerpo técnico" lista al staff del equipo.
- Click en un ayudante abre `/equipos/[teamId]/staff/[membershipId]/capabilities`.
- Panel con las 8 capabilities como toggles (switch) con etiqueta y descripción corta.
- Toggle individual = UPDATE optimista con server action; revierte si la action falla.
- Mensajes claros de error si el user no tiene permiso (RLS lo rechaza).

## 3. Fuera de alcance

- Capabilities granulares por equipo: hoy las caps son por membership (por club), no por equipo. Si en un futuro un principal del equipo Alevín A no debería poder modificar las caps del ayudante del Cadete B, eso pide refactor del modelo. **No se aborda aquí**.
- Histórico de cambios de caps (audit log llegará en F14).
- Caps personalizadas (texto libre). Las 8 son las definidas en F1.4. Cambiar la lista pide nueva migración + extender CHECK constraint.
- Aplicar caps a otros roles (jugador, coordinador): hoy solo entrenador_ayudante tiene caps.

## 4. Modelo de datos afectado

**Cero SQL nuevo**. Reusa `capabilities`, `memberships`, helpers `user_role_in_club`, `user_has_capability_in_club`.

Verificación previa a este lote: comprobar en `pnpm db:test` que las RLS de `capabilities_select` y `capabilities_update` son las esperadas (admin/coord/principal del club). Si en algún momento se quiere endurecer ("solo principal del MISMO equipo del ayudante puede tocar"), eso es un refactor del modelo (necesita `team_id` en la membership o tabla puente nueva). No se hace ahora — anotado como mejora futura en §8.

## 5. UI

### Estructura de rutas

```
apps/web/src/app/[locale]/(authenticated)/
├── equipos/
│   ├── [teamId]/
│   │   ├── page.tsx                                       ← Detalle de equipo (F2.6 introduce el listado de staff)
│   │   └── staff/
│   │       └── [membershipId]/
│   │           └── capabilities/
│   │               ├── page.tsx                           ← Panel toggles
│   │               ├── capability-toggle.tsx              ← Componente client
│   │               └── actions.ts                         ← Server action updateCapability
```

### Componentes

- `<CapabilitiesPanel>` (server): carga el listado de capabilities del membership ordenadas por nombre canónico. Verifica server-side que el user es admin/coord/principal del club. Si no, `notFound()` o redirect a `/equipos/[teamId]`.
- `<CapabilityToggle>` (client): un `<input type="checkbox" role="switch">` (o `Switch` de shadcn si se introduce — ver §8). Optimistic update con `useOptimistic` o estado local + acción.

### Layout de la pantalla

Header: "{nombre del ayudante} · {nombre del equipo}".
Subtítulo: link de vuelta al equipo.
Lista de 8 filas, cada una con:

- Toggle (granted true/false).
- Etiqueta i18n (`capabilities.can_evaluate.label`).
- Descripción corta i18n (`capabilities.can_evaluate.description`).

Al final: mensaje "Cambios se guardan automáticamente. Última actualización hace N minutos."

## 6. Estados, validaciones y errores

### Esquema Zod nuevo en `packages/core/src/schemas/capabilities.ts`

```ts
export const updateCapabilitySchema = z.object({
  membership_id: z.string().uuid(),
  capability_name: z.enum([
    'can_evaluate',
    'can_create_lineups',
    'can_register_match_events',
    'can_create_sessions',
    'can_create_plays',
    'can_see_medical',
    'can_message_families',
    'can_manage_squad',
  ]),
  granted: z.boolean(),
});
```

### Estados de UI

- **Carga inicial**: server-renderiza con los valores actuales. Sin spinner.
- **Toggle en flight**: el switch queda visualmente "pending" (opacity 60% o pequeño spinner). Si la action falla, revierte y muestra toast i18n con el error.
- **Membership no es ayudante**: la página muestra mensaje "Este miembro no es entrenador ayudante. Las capabilities solo aplican a ayudantes." y no pinta toggles. El check se hace server-side leyendo `memberships.role`.
- **User sin permiso**: la server action devuelve `{ success: false, error: 'forbidden' }`; cliente revierte y pinta toast. La RLS es la autoridad real (defense in depth).
- **Capability inexistente**: imposible si el trigger funcionó. Si por alguna razón faltase una fila, la action hace `upsert` en lugar de `update`. Anotado en §8.

### Errores tipados

```ts
type CapabilityActionResult =
  | { success: true; granted: boolean }
  | { success: false; error: 'forbidden' | 'not_assistant' | 'invalid_input' | 'db' };
```

- **Unit (Vitest)** — pospuesto (sin infra). Schemas via typecheck.
- **RLS (pgTAP)** — añadir `supabase/tests/rls_capabilities_update.sql` que cubra:
  1. Admin del club A puede UPDATE de caps de ayudante en club A.
  2. Coord del club A puede UPDATE.
  3. Principal del club A puede UPDATE.
  4. Ayudante NO puede UPDATE de sus propias caps.
  5. Jugador del club A NO puede UPDATE.
  6. User del club B NO puede UPDATE de caps del club A.
  7. UPDATE de una capability_name inválida falla por CHECK constraint.

## 8. Notas de implementación

- **shadcn `Switch` o checkbox manual**: decide al implementar. Si se introduce shadcn en este lote por primera vez, abre ADR-0005 documentándolo y aplicando el setup (init + `@/components/ui`). Si se prefiere mantener Tailwind manual, basta con un `<button role="switch" aria-checked={granted}>` accesible.
- **UPSERT en lugar de UPDATE**: por robustez, la action hace `.upsert({ membership_id, capability_name, granted })` con `onConflict: 'membership_id,capability_name'`. Cubre el caso (poco probable) de que falte la fila sembrada por el trigger.
- **Limitación de RLS conocida**: hoy un entrenador principal del club puede tocar las caps de cualquier ayudante del club, no solo del equipo donde es principal. Esto era explícito en F1.7 (las policies son intencionadamente permisivas). Cuando el producto tenga más de un equipo por club operando con principales distintos, conviene endurecer. Lo dejo registrado en `known-issues.md` como "F2.7: capabilities cross-team — endurecer RLS cuando haya datos multi-equipo activos" → resolver en F11 o cuando aparezca.
- **i18n**: 8 × 2 strings (label + description) en `es/en/va` bajo `capabilities.<name>.label|description`. Texto fuente en `es`; `en/va` espejan hasta F14.
- **Optimistic UX**: `useOptimistic` de React 19 funciona bien aquí. Caer a estado local si genera complejidad innecesaria.
