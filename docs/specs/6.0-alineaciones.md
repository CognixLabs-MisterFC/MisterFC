## 2. Decisión 1 — Modelo de datos: normalizado (Opción A)

**Decisión: Opción A — tablas normalizadas `lineups` + `lineup_positions`.** Rechazada la Opción B (blob JSON).

### Justificación (editabilidad · queryability F9 · simplicidad)

1. **Queryability futura (F9 stats posicionales)** — peso decisivo. El plan-maestro cita explícitamente "estadísticas posicionales por jugador en F9". Esas consultas ("minutos por posición", "nº de veces titular como lateral derecho") son un `GROUP BY player_id, position_code` trivial e indexable sobre filas. Con blob JSON habría que desempaquetar el jsonb de cada lineup en cada query (`jsonb_array_elements`, sin índices útiles, full scans) — coste creciente con el histórico de partidos.
2. **Editabilidad** — el drag&drop bidireccional campo↔banquillo↔fuera (F6.7) muta el estado de **un** jugador por gesto. Normalizado = un `UPDATE` de fila por drop. Blob = reescribir todo el documento en cada drop, con riesgo de lost-update si dos pestañas editan, y sin posibilidad de RLS/constraint a nivel jugador.
3. **Integridad referencial** — FK `player_id → players`, FK formación → catálogo. Un jugador dado de baja deja el blob con basura silenciosa; con FK + `on delete` el modelo se mantiene coherente.
4. **Constraints declarativos** — el invariante duro "un jugador en una sola zona" (F6.7) es un `unique (lineup_id, player_id)` en normalizado, garantizado por la BD. En blob es lógica de app sin red de seguridad.
5. **Coherencia con las subfases ya diseñadas** — 6.7 (`location`/`out_reason`), 6.8 (`planned_substitutions`) y 6.9 (`lineup_player_notes`, `lineup_objectives`, `lineup_phase_notes`) ya están pensadas como columnas/tablas relacionales. El blob obligaría a un híbrido inconsistente (parte en JSON, parte en tablas).

**Trade-off aceptado**: el normalizado cuesta más migraciones ahora. Compensa por F9 + editabilidad + integridad. → **ADR-0012** (§11).

### Modelo

```sql
-- Cabecera de alineación. Varias por partido (F6.4).
create table public.lineups (
  id              uuid primary key default gen_random_uuid(),
  event_id        uuid not null references public.events(id) on delete cascade,
  name            text not null check (char_length(name) between 1 and 60),  -- "Titular", "Plan B", "2ª parte"
  formation_code  text not null,                 -- ref. al catálogo en código (§3); p.ej. '1-3-3', '4-3-3'
  is_official     boolean not null default false,
  visibility      text not null default 'staff' check (visibility in ('staff', 'team')),  -- §5
  tactical_notes  text check (tactical_notes is null or char_length(tactical_notes) <= 2000),  -- F6.9
  created_by      uuid not null references public.profiles(id),
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now()
);

-- Una sola alineación oficial por partido.
create unique index lineups_one_official_per_event
  on public.lineups (event_id) where is_official;

create index lineups_event_idx on public.lineups (event_id);

-- Posición de cada jugador en una alineación. Cubre TODO el roster convocado:
-- field (titular) | bench (suplente) | out (fuera de convocatoria).
create table public.lineup_positions (
  id              uuid primary key default gen_random_uuid(),
  lineup_id       uuid not null references public.lineups(id) on delete cascade,
  player_id       uuid not null references public.players(id),
  location        text not null default 'bench' check (location in ('field', 'bench', 'out')),
  position_code   text,             -- slot del preset (solo field); p.ej. 'GK','DC','MC1'…
  x_pct           numeric(5,2) check (x_pct is null or (x_pct >= 0 and x_pct <= 100)),  -- snap, solo field
  y_pct           numeric(5,2) check (y_pct is null or (y_pct >= 0 and y_pct <= 100)),
  callup_status   text check (callup_status is null or callup_status in (   -- snapshot F6.6
    'disponible','duda','no_disponible','descarte_tecnico','sin_dato')),
  out_reason      text check (out_reason is null or out_reason in (         -- solo out (F6.7)
    'tecnico','fisico','disciplinario','personal')),
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),

  unique (lineup_id, player_id),                         -- un jugador, una zona

  constraint lineup_positions_field_has_position check (
    (location = 'field' and position_code is not null)
    or (location in ('bench','out') and position_code is null)),

  constraint lineup_positions_out_reason_coherent check (
    (location = 'out'  and out_reason is not null)
    or (location <> 'out' and out_reason is null))
);

create index lineup_positions_lineup_idx on public.lineup_positions (lineup_id);
create index lineup_positions_player_idx on public.lineup_positions (player_id);  -- F9
```

⚠ **Reconciliación con bosquejo 6.6**: el bosquejo 6.6 dudaba entre columna en `lineup_positions` vs tabla `lineup_callup_snapshot` aparte, "probable B porque lineup_positions solo carga titulares". **Esa premisa cambió**: con el modelo `location` de 6.7, `lineup_positions` ya contiene field+bench+out = **todo el roster convocado**, así que `callup_status` vive como **columna en `lineup_positions`** (no tabla aparte). Un solo lugar, una sola RLS.

Las tablas auxiliares de 6.8 (`planned_substitutions`) y 6.9 (`lineup_objectives`, `lineup_player_notes`, `lineup_phase_notes`) se mantienen como en sus bosquejos (§7, §8 de esta spec).

## 3. Decisión 2 — Formaciones: catálogo en código + modalidad = `teams.format`

**Decisión A — catálogo de formaciones como const tipada en `packages/core` (NO tabla en BD).**
**Decisión B — la modalidad es `teams.format` (ya existe); NO se infiere de categoría/half_duration ni se añade `team.match_modality`.**

### Justificación catálogo en código

- Las formaciones son **datos de referencia estáticos**: cambian con un release, no por club ni por temporada. No hay CRUD, ni RLS, ni variación multi-tenant en Ola 1.
- El único caso que justificaría tabla — "plantillas de formación guardadas por entrenador" — está **explícitamente fuera de alcance de F6** (futuro). Hasta entonces una tabla es overhead puro (migración + seed + RLS + roundtrip de red por cada render del editor).
- En `packages/core` (agnóstico de framework, reutilizable por `apps/native` en Ola 2): acceso tipado, sin red, validado por Zod. `lineups.formation_code` guarda el id por string; si una formación se retira del catálogo, el código guardado sigue renderizando (degradación elegante: se muestra el code crudo).
- → **ADR-0013** (§11).

### Justificación modalidad = `teams.format`

- `teams.format` **ya existe** con CHECK `('F7','F8','F11')` desde F1.1. Es la fuente explícita y autoritativa. Añadir `team.match_modality` sería duplicar.
- `half_duration_minutes` es **age-based** (querubín 15 … amateur 45), **no** format-based. Un alevín puede jugar F7 u F8; un infantil F11. **Inferir la modalidad de la duración daría resultados incorrectos.** Por eso no se infiere.
- El editor obtiene el formato vía `event → team → format`.

### Catálogo propuesto (`packages/core/src/lineups/formations.ts`)

Cada formación: `{ code, label, format, slots: [{ code, role, x_pct, y_pct }] }`. `x_pct/y_pct` en sistema 0–100 con el equipo atacando hacia arriba (portero abajo). Listado inicial común del fútbol base español:

- **F7** (6 de campo + portero): `1-3-3` (3 def, 3 del por delante: clásica), `1-3-2-1` (rombo), `1-2-3-1`, `1-2-1-2-1`.
- **F8** (7 de campo + portero, variante regional Canarias): `1-3-3-1`, `1-3-1-2-1`, `1-2-3-2`, `1-4-2-1`.
- **F11**: `4-4-2`, `4-3-3`, `4-2-3-1`, `3-5-2`, `4-4-2 rombo`, `5-3-2`, `3-4-3`.

`role` ∈ {`GK`,`DF`,`MF`,`FW`} para mapear sugerencias con `players.position_main` al importar (snap automático del portero a GK, etc.).

## 4. Decisión 3 — `<MatchFieldEditor>` compartido F6/F7

**Decisión: un único componente con prop `mode`, NO componente duplicado ni dos variantes.** Alineado con [ADR-0009](../decisions/ADR-0009-f6-f7-match-field-editor-compartido.md) (ya **Accepted**).

### API (diseñada en F6.3 con F7 ya en mente)

```tsx
type FieldMode = 'edit' | 'readonly' | 'live-overlay';

interface MatchFieldEditorProps {
  format: 'F7' | 'F8' | 'F11';
  formationCode: string;
  positions: FieldPlayer[];              // {playerId, location, positionCode?, xPct?, yPct?, label, dorsal}
  mode: FieldMode;                       // edit (F6) | readonly (familia/F6.9 vista) | live-overlay (F7)
  onPlayerMove?: (playerId, toLocation, toSlot?) => void;   // F6 edit
  onPlayerClick?: (playerId) => void;    // F7 (asignar evento a jugador)
  onFieldClick?: (xPct, yPct) => void;   // F7 (evento sobre el césped)
  onPlayerHover?: (playerId | null) => void;
  children?: React.ReactNode;            // overlays absolutos (cronómetro/paleta de F7)
}
```

- **Sin lógica de "eventos de partido" dentro** — el componente solo conoce jugadores, posiciones y zonas. F7 superpone su capa vía `children` + los callbacks `onPlayerClick`/`onFieldClick`.
- `mode='readonly'` cubre la visualización para familias/jugadores (§5) y la vista previa de 6.9.
- **Ubicación**: el componente React vive en `apps/web/src/components/match/match-field-editor.tsx` (`packages/core` es agnóstico de React). La **geometría pura** (resolver slots de una formación, snap de coordenadas, reasignación al cambiar de formación) vive en `packages/core/src/lineups/` como funciones puras → reutilizable por `apps/native` (Ola 2) y testeable con Vitest sin DOM.
- **Cambio de formación** (F6.3): función pura `remapPositions(prev, nextFormation)` que conserva asignaciones por `role` cuando es posible (un DF se queda en un slot DF del nuevo preset) y manda al banquillo los que sobran. Testeable aislada.

### Librería drag&drop

**Recomendación: `@dnd-kit/core` + `@dnd-kit/sortable`.** Motivos: soporte táctil (crítico para F7 en iPad apaisado), accesible (teclado), tree-shakeable, sin `findDOMNode` (compatible React 19), mantenida. Alternativa descartada: `react-dnd` (HTML5 backend flojo en táctil, requiere backend touch aparte). Es una **dependencia de librería, no una credencial** → no requiere aviso previo. Se añade en Lote A con la subfase 6.3.

## 5. Decisión 4 — Permisos

### Escritura (crear/editar alineación): reutilizar `can_create_lineups`

**Decisión: reutilizar la capability existente `can_create_lineups` (F1.4). NO crear capability nueva.**

Justificación: a diferencia de F4 (que sí creó `can_manage_callups` porque "gestionar convocatoria" no encajaba en ninguna capability previa), aquí **ya existe** la capability semánticamente exacta. Crear `can_manage_lineup` sería duplicar y rompería el seed/backfill de F1.4. ⚠ Se reconcilian los bosquejos 6.6/6.7/6.9 (que la nombraban mal) a `can_create_lineups`.

Helper nuevo, espejo de `user_can_manage_callup` (incluyendo la lección "autoridad por equipo = `team_staff.staff_role`, no `memberships.role`", fix F4 Lote B):

```sql
create or replace function public.user_can_manage_lineup(p_event_id uuid)
returns boolean language sql stable security definer set search_path = public as $$
  select
    public.user_role_in_club(e.club_id) in ('admin_club','coordinador')
    or (e.team_id is not null and exists (
      select 1 from public.team_staff ts
      join public.memberships m on m.id = ts.membership_id
      where ts.team_id = e.team_id and ts.staff_role = 'entrenador_principal'
        and ts.left_at is null and m.profile_id = auth.uid() and m.club_id = e.club_id))
    or (e.team_id is not null
        and public.user_has_capability_in_club(e.club_id, 'can_create_lineups')
        and public.user_is_staff_of_team(e.team_id))
    from public.events e where e.id = p_event_id;
$$;
```

### Lectura: decisión de producto — flag `visibility` por alineación

**Decisión: cada alineación tiene `visibility ∈ {staff, team}` (default `staff`). El cuerpo técnico decide. Solo la alineación `is_official` con `visibility='team'` es visible a jugadores/familias.**

Justificación de producto: en fútbol base/amateur conviven dos prácticas legítimas — entrenadores que anuncian el once para motivar, y entrenadores que lo mantienen privado hasta el día del partido (evitar quejas, cambios de última hora). Un flag sirve a ambos; el **default privado (`staff`)** respeta el caso conservador y evita filtrar el once antes de que el técnico esté listo. No se ata a "antes/después del pitido": el técnico publica cuando quiere.

RLS `lineups`:
- SELECT: staff con `user_can_manage_lineup` siempre; cualquier miembro del club solo si `is_official AND visibility='team'`.
- INSERT/UPDATE/DELETE: `user_can_manage_lineup`.
- `lineup_positions` hereda (lectura ligada al `lineup_id` legible).
- **Notas tácticas y objetivos (6.9) son staff-only siempre**, independientes del flag (tabla/columna con su propia RLS). Notas individuales por jugador solo si `visible_to_player` (ver §8).

## 6. Decisión 5 — Integración con F4 (convocatoria)

**Decisiones:**

1. **La alineación se puede crear ANTES de cerrar la convocatoria.** El lineup es el artefacto de planificación del técnico; bloquearlo hasta cerrar convocatoria impediría planificar con antelación. Los jugadores aún sin responder aparecen como `sin_dato` (badge "pendiente") en el editor.
2. **El import (F6.6) requiere convocatoria PUBLICADA** — un `draft` no expone respuestas (regla heredada de F4). Sin convocatoria publicada, el técnico aún puede montar la alineación **manualmente** desde el roster (`team_members` histórico a la fecha), sin enriquecimiento de `callup_status`.
3. **Seleccionables como titular/suplente**: por defecto los confirmados (`yes`/`maybe` y no descartados). El editor **permite override** del técnico (arrastrar un `pendiente`/`sin_dato` al campo) marcándolo visualmente como pendiente — el técnico manda, la app avisa pero no bloquea.
4. **El import es un snapshot** (no lectura viva). Cambios posteriores en F4 no mutan el plan en silencio; el técnico **reimporta explícitamente** (botón) mientras el partido sea futuro. Mapeo `(response, decision) → callup_status` (testeable puro):

   | response \ decision | called_up | discarded | (sin decisión) |
   |---|---|---|---|
   | yes | disponible | descarte_tecnico | disponible |
   | maybe | duda | descarte_tecnico | duda |
   | no | no_disponible | descarte_tecnico | no_disponible |
   | (sin respuesta) | disponible | descarte_tecnico | sin_dato |

   `descarte_tecnico` tiene prioridad sobre la respuesta del jugador. Al importar: `descarte_tecnico`/`no_disponible` → `location='out'` (con `out_reason='tecnico'` por defecto en descarte, editable); resto → `location='bench'` para que el técnico los arrastre al campo.

## 7. Decisión 6 — Subfases 6.6–6.9 (interpretación) y §8 modelos auxiliares

Interpretación consolidada (los bosquejos individuales siguen siendo la referencia de detalle; aquí la lectura actual):

- **6.6 Importar convocatoria** — CTA "Importar convocatoria" que genera el snapshot `callup_status` por jugador del roster (mapeo de §6) + botón "Reimportar". Lectura unidireccional F4→F6. Sin convocatoria publicada: fallback manual desde roster.
- **6.7 Banquillo + drag&drop bidireccional** — panel(es) "Banquillo" y "Fuera de convocatoria" como zonas de drop; movimiento campo↔banquillo↔fuera con invariante "un jugador, una zona" (unique index) y `out_reason` obligatorio al sacar. Contadores por zona. **Es el corazón del editor** — sin esto, 6.3 entrega medio editor.
- **6.8 Cambios programados** — tabla `planned_substitutions` (minuto / sale / entra / razón / notas), lista ordenada por minuto, validación de coherencia (out en campo/banquillo, in en banquillo) vía trigger, fila en rojo si referencia a jugador ya movido. Es un **plan**, no se ejecuta (F7 lo consume).
- **6.9 Notas tácticas** — `lineups.tactical_notes` (≤2000) + `lineup_objectives` (≤6, checklist con `achieved`) + `lineup_player_notes` (≤1/jugador, toggle `visible_to_player`) + `lineup_phase_notes` (≤4 fases). Staff-only salvo notas individuales con `visible_to_player`.

## 8. Decisión 7 — Lote split (recomendado)

**Recomendación: SPLIT en 2 lotes.** Un PR único de 9–14 h es demasiado para revisar bien (feedback del responsable: reviewability + 3 checkpoints por fase). Frontera elegida: **"editor plenamente usable por el staff" (A) vs "integración F4 + planificación + visibilidad" (B)**.

### Lote A — Fundación + editor del staff (~7–9 h)
- 6.1 modelo (`lineups`, `lineup_positions` con `location`) + helper `user_can_manage_lineup` + RLS.
- 6.2 catálogo de formaciones (`packages/core`) + geometría pura.
- 6.3 `<MatchFieldEditor>` + drag&drop (dnd-kit) + cambio de formación.
- 6.4 múltiples alineaciones por partido + marcar oficial.
- 6.5 lista "fuera de convocatoria" con motivo.
- 6.7 banquillo + drag&drop bidireccional (**va en A**: el editor está incompleto sin las 3 zonas; separarlo dejaría medio editor sin demostrar).
- **Deliverable verificable**: el técnico crea una alineación, monta titular+banquillo+fuera vía drag&drop, varias por partido, marca la oficial, persiste.

### Lote B — Integración F4 + planificación + visibilidad (~4–6 h)
- 6.6 importar convocatoria (snapshot + reimport) + fallback manual.
- 6.8 cambios programados (`planned_substitutions` + trigger de coherencia).
- 6.9 notas tácticas (notas/objetivos/individuales/fases).
- `visibility` flag + vistas `readonly` para jugador/familia (reuso de `<MatchFieldEditor mode='readonly'>`).
- **Deliverable verificable**: el técnico importa la convocatoria, planifica cambios, escribe notas, publica el once al equipo; la familia lo ve.
