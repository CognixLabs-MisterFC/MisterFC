# Spec 6.9 — Notas tácticas del partido

> Subfase del Plan Maestro: **6.9 — Notas tácticas: bloque libre + objetivos + indicaciones por jugador o por fase**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-29 (bosquejo)

---

## 1. Contexto

La preparación de un partido incluye decisiones tácticas que no caben en una formación dibujada: presión arriba o repliegue, jugador clave del rival a vigilar, instrucciones individuales (al lateral derecho: "no acompañar al extremo en la segunda parte"), objetivos por fases (primer cuarto = aguantar; último cuarto = arriesgar atrás). Esta subfase añade ese tejido textual a la alineación.

## 2. Objetivos

- **Notas generales del partido**: textarea libre (≤ 2000 caracteres) editable por el staff.
- **Objetivos**: lista corta (≤ 6 ítems) tipo checklist (`{texto, lograda boolean nullable}`). Se marcan en F7 al cierre del partido o en F8 al hacer la valoración.
- **Indicaciones por jugador**: ≤ 1 nota por jugador del lineup (≤ 300 caracteres). Solo visible para el staff y opcionalmente para el propio jugador (toggle de visibilidad por nota — útil cuando es info sensible que no quieres exponer al equipo entero).
- **Indicaciones por fase del partido**: ≤ 4 fases (primer cuarto / segundo cuarto / descanso / tercer cuarto / etc.; granularidad configurable). Cada fase con su texto corto.

## 3. Fuera de alcance

- **NO** se generan indicaciones por IA / heurísticas (basadas en stats del rival, evolución del jugador). Esa pieza es de **Ola 3** como mínimo.
- **NO** se traduce al jugador automáticamente. Si la nota es individual + visible, la ve tal cual la escribió el staff.
- **NO** se trackea histórico de versiones de la nota. La edición sobrescribe.
- **NO** se vincula con los datos en directo de F7 ("la indicación se cumplió?"). El cruce indicación → ejecución es Ola posterior.

## 4. Modelo de datos afectado

```sql
-- Notas generales + objetivos sobre el lineup.
alter table public.lineups
  add column tactical_notes text
    check (tactical_notes is null or char_length(tactical_notes) <= 2000);

create table public.lineup_objectives (
  id          uuid primary key default gen_random_uuid(),
  lineup_id   uuid not null references public.lineups(id) on delete cascade,
  position    smallint not null check (position between 1 and 6),
  text        text not null check (char_length(text) between 1 and 200),
  achieved    boolean,
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now(),
  unique (lineup_id, position)
);

-- Indicaciones por jugador.
create table public.lineup_player_notes (
  lineup_id           uuid not null references public.lineups(id) on delete cascade,
  player_id           uuid not null references public.players(id),
  note                text not null check (char_length(note) between 1 and 300),
  visible_to_player   boolean not null default false,
  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now(),
  primary key (lineup_id, player_id)
);

-- Indicaciones por fase.
create table public.lineup_phase_notes (
  lineup_id    uuid not null references public.lineups(id) on delete cascade,
  phase_index  smallint not null check (phase_index between 1 and 4),
  phase_label  text not null check (char_length(phase_label) between 1 and 80),
  note         text not null check (char_length(note) between 1 and 600),
  primary key (lineup_id, phase_index)
);
```

RLS:
- `lineups.tactical_notes` / `lineup_objectives` / `lineup_phase_notes`: lectura para el staff del team_staff. Jugador NO ve.
- `lineup_player_notes`: lectura por staff del team_staff + lectura por jugador si `visible_to_player = true` AND tiene `player_accounts` con `relation='self'` para `player_id`.

## 5. UX

- Tab "Notas" en el editor de F6 (junto a "Formación", "Banquillo", "Cambios").
- Secciones: Notas generales / Objetivos / Indicaciones individuales (lista de jugadores del lineup con icono "tiene nota") / Por fase (4 acordeones).
- Toggle "visible al jugador" como switch shadcn por nota individual.

## 6. Permisos

- Escritura: principal del `team_staff` o `can_manage_lineup`.
- Lectura: ver §4 RLS por tabla.

## 7. Tests

- pgTAP:
  - Notas generales: principal escribe, jugador no.
  - `lineup_player_notes` con `visible_to_player=true`: el dueño del `player_account` (relation=self) lee; con `false`, no.
  - Familia (relation=parent) lee siempre las notas individuales de su hijo, independiente del flag (decisión heredada del modelo F4: familia ve lo del jugador).
- Vitest: serialización de objetivos / fases / notas para mostrar en F7.
- Smoke: escribir 3 objetivos, marcar el segundo como `achieved=true` al cerrar partido en F8.
