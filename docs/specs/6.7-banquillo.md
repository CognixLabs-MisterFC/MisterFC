# Spec 6.7 — Banquillo del partido + drag&drop bidireccional

> Subfase del Plan Maestro: **6.7 — Banquillo del partido: titulares + reservas + fuera convocatoria con drag&drop bidireccional**.
> Estado: ☐ pendiente
> Autor: Iker Milla
> Fecha de creación: 2026-05-29 (bosquejo)

---

## 1. Contexto

F6.3 entrega el editor de campo con drag&drop de jugadores convocados sobre las posiciones de la formación. Pero un partido no se planifica solo con el 11 inicial: el coach decide explícitamente quién está en el banquillo (potenciales sustitutos), quién queda fuera de convocatoria (con motivo), y reorganiza esa lista a mano sobre la marcha. Esta subfase añade el panel de banquillo como segunda zona de drop con drag&drop bidireccional al campo.

## 2. Objetivos

- Panel lateral o inferior que muestra **2 listas verticales**: "Banquillo" y "Fuera de convocatoria".
- **Drag&drop bidireccional**: arrastrar un jugador desde el banquillo al campo lo coloca en una posición de titular y deja el slot del campo anterior libre. Lo simétrico al revés.
- Arrastrar un jugador del campo al "Fuera de convocatoria" lo saca del partido (precisa motivo: `tecnico` / `fisico` / `disciplinario` / `personal`).
- **Restricción dura**: un jugador no puede estar en más de un sitio a la vez. Las tres zonas (campo, banquillo, fuera) son mutuamente excluyentes.
- Contador en cada zona ("Banquillo · 5 jugadores", "Fuera convocatoria · 2") visible siempre.

## 3. Fuera de alcance

- **NO** se simula visualmente un cambio entre titular y banquillo en el editor (animación de sustitución durante el partido). Eso es F7 (modo "planning preview" si se quisiera, no comprometido).
- **NO** se valida el límite reglamentario de jugadores por banquillo según `format` (F7 / F8 / F11). Solo se cuenta y se muestra. Las reglas de competición las pone el coach manualmente o se añaden en una Ola posterior.
- **NO** se exporta el banquillo a un acta arbitral. Esa pieza es F16 (operativa beta) si surge.

## 4. Modelo de datos afectado

Dos opciones:

**Propuesta A — columna `location` en `lineup_positions`** (recomendada):

```sql
alter table public.lineup_positions
  add column location text not null default 'field'
    check (location in ('field', 'bench', 'out')),
  add column out_reason text
    check (
      (location <> 'out' and out_reason is null)
      or (location = 'out' and out_reason in ('tecnico', 'fisico', 'disciplinario', 'personal'))
    );

-- Para 'bench' y 'out', position_id queda nullable.
alter table public.lineup_positions
  alter column position_id drop not null;

alter table public.lineup_positions
  add constraint lineup_positions_field_has_position
    check (
      (location = 'field' and position_id is not null)
      or (location in ('bench', 'out') and position_id is null)
    );
```

**Propuesta B — tabla `bench_positions` separada**:
- Más limpia conceptualmente pero duplica policies/triggers.
- Tres tablas (`lineup_field_positions`, `bench_positions`, `out_positions`) o una con `location`.

**Decisión recomendada**: Propuesta A. Una sola tabla, una sola RLS, un solo trigger de validación. La columna `location` separa el rol del jugador en la alineación; `position_id` solo aplica al campo.

## 5. UX

- Layout 3 columnas o stack vertical en móvil:
  - Izquierda: panel "Banquillo" (lista vertical con chip de jugador).
  - Centro: campo SVG (`<MatchFieldEditor>`).
  - Derecha: panel "Fuera de convocatoria".
- Drag&drop con `dnd-kit` o equivalente (a decidir; F6.3 ya habrá elegido).
- Sobre drop del campo a fuera: modal pidiendo motivo.
- Sobre drop de fuera a campo/banquillo: motivo se borra automáticamente.

## 6. Permisos

- Edita el banquillo: `can_manage_lineup` (F6.1) o principal del `team_staff` del partido.
- Lee: cualquier miembro del club según RLS de `lineups` (decisión heredada de F6.1).

## 7. Tests

- Vitest puro del reducer del editor: arrastrar A del campo al banquillo libera el slot del campo y añade A al banquillo. Invariante "un jugador en una zona" siempre se cumple tras cualquier secuencia de drops.
- pgTAP: CHECK constraint `lineup_positions_field_has_position` falla con `location='field'` y `position_id=null`.
- Smoke: 11 titulares + 5 banquillo + 2 fuera, total 18 jugadores. Mover uno del banquillo al campo desplaza al titular sustituido al banquillo (no se duplica).
